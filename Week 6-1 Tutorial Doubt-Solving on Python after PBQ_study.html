Python Doubt-Solving — the Mechanics

Week 6-1 · Tutorial Session (instructor Mahavir) — a practical follow-up to PBQ-01, written as a lesson you can read end to end.

The first Python lecture taught you the building blocks; this tutorial clears up the small mechanical things that quietly trip up every beginner: how importing actually works, what a working directory is, why one method needs parentheses and another doesn't, and which of the three brackets to reach for. None of it is hard once you see the logic — and the logic is what this guide gives you.

1. What a library really is

A library is pre-assembled code bundled for a specific job. The instructor's analogy is a set of labelled toolboxes at home: one box for home repairs (hammer, spanner, bolts), one for grooming (razor, nail clipper), one for electronics (cables, drives). You don't groom with a hammer, and you don't analyse data with a plotting library. Each Python library has its lane:

LibraryName originJob
pandaspanel dataread, clean, manipulate and analyse tabular data
numpynumerical Pythonfast numeric and matrix mathematics
matplotlibmath plotting librarycharts and figures

There are roughly 137,000 libraries published on PyPI — for astronomy, chemistry, machine learning, anything — but in quant finance you'll lean on maybe 15–20, and use 10–12 daily. Don't be overwhelmed by the size of the ecosystem; you only ever load the handful you need.

Built-in vs third-party

Some libraries ship with Python and need no installation — os, urllib, math. These are built-in, like the apps pre-loaded on a new phone. Others — numpy, pandas, sklearn — are third-party: maintained by an outside community, hosted on a server, and you must install them once before you can use them (like downloading Uber or a food-delivery app yourself).

Install once, import every session

These two words confuse beginners constantly. Installing downloads the library from its server to your hard drive — you do this once. Importing loads it into memory so your current program can use it — you do this every time you start Python or restart the kernel.

The cupboard analogy: you buy a toolkit once and store it in a cupboard (install). Each time you need it you fetch it out, use it, and put it back (import). Buy once; fetch on every use.

2. The working directory

When you write pd.read_csv("data.csv"), Python looks for that file in your current working directory (CWD). If the file lives elsewhere — say, in Downloads — you get a FileNotFoundError. The os library is the fix:

CommandWhat it does
os.getcwd()shows the directory you're currently in
os.chdir(path)moves you to a new directory
os.listdir()lists the files there
dir(library)lists everything inside a library (handy when you forget a method name)

So you have two ways to read a file outside your CWD: copy the file into the working directory, or change the working directory to where the file is. (On Windows, remember paths in code need double backslashes, C:\\\\Users\\\\..., or the single backslash gets read as an escape character.)

3. The five ways to import — and the order trap

There's more than one way to bring a library in, each with a use case:

FormWhen to use it
import mathplain and simple
import numpy as nppreferred — the alias saves you typing the full name repeatedly
from math import *discouraged — pulls in everything and floods the namespace
from math import sqrtgrab one function — good for heavy libraries like sklearn or pytorch
from math import sqrt as rootone function, renamed

A common beginner slip is the word order. It is from B import A, never import A from B. The reasoning is intuitive: the library (B) is the thing stored on disk, so Python looks it up first and then pulls the function (A) out of it.

4. Namespace overriding — why import * is risky

Here's the concrete danger of star-imports. Suppose two libraries both define sin:

StepCodeEffect
1from math import sinreal-valued sine loaded
2from cmath import sincomplex-valued sine overrides the first
resultsin(1.0)returns (0.841+0j) — the cmath version won

Whatever you import later wins. In a small script that's a curiosity; in a large project with many files and variables, it becomes impossible to trace where a function came from. The cure is to keep names explicit — write math.sin(1.0) — so any reader knows exactly which library each call belongs to.

5. Attribute vs method: the parentheses puzzle

This was the tutorial's headline question, raised by a student who couldn't see why df.shape works but df.shape() errors. The distinction is simple once named:

  • An attribute is a stored value you just read — df.shape returns a tuple like (2460, 7). No parentheses.
  • A method is an action you call — df.describe() computes statistics. It needs parentheses, because the parentheses are where arguments go.

Call an attribute as if it were a method and Python complains "'tuple' object is not callable." The general syntax to remember is library.method(arguments) — if there are no arguments to pass and you're simply reading a fact, you're looking at an attribute, and the parentheses disappear.

A useful tell: in Jupyter, type the name and check its type. type(df.shape) is tuple (a value); type(df.describe) is method (a callable).

6. Which bracket goes where?

Three brackets, three jobs. This is worth committing to memory because mixing them up is the single most common syntax error for newcomers.

BracketUsed forExample
round ( )call functions/methods, pass arguments, define tuples, group arithmeticdef f(a,b): return (a+b, a*b)
square [ ]define lists, index and slicelst[0], lst[0:3]
curly { }define sets and dictionaries, hold variables in f-strings{"MSFT":300}, f"{price}"

Recall the key behavioural differences too: a list (square brackets) is mutable — you can .append to it — while a tuple (round brackets) is immutable, so it has no .append at all. A set (curly brackets) keeps only unique elements and is unordered; a dictionary (also curly) stores key–value pairs. Two small punctuation marks round this out: the comma separates items and arguments just like in English, and the underscore makes long variable names readable (my_first_python_variable).

7. f-strings: cleaner output

You met f-strings in PBQ-01; the tutorial reinforced why they matter. In real code you often print status messages built from several variables and conditions — "since condition X was met, we chose strategy A." An f-string (prefixed with f, variables wrapped in curly braces) makes those messages readable without clumsy string concatenation. Readable output is not a luxury; in a live trading algorithm it is how you know what your code just decided.

8. Magic commands and the lesson of %timeit

Magic commands are Jupyter shortcuts that begin with %. The everyday ones:

CommandWhat it shows
%whoa bare list of your defined variables
%whosvariables with their types and values (more organised)
%pwdthe current working directory
%historyevery command run since the kernel started
%timeitbenchmarks a line by running it many times, reporting mean ± std-dev

The deepest point came from %timeit. Benchmarking sum(range(n)) for growing n shows that work does not scale linearly: summing 10 numbers took ~139 ns, summing 20 took ~184 ns (not double — fixed call overhead dominates at small sizes), and summing 1,000 took ~10 µs. Looking at the ratio of timings as the input grows is how you gauge a routine's time complexity and choose the more scalable of two approaches.

On using AI to write code: the instructor encourages it — but with a caveat. If an LLM hands you a strategy using rolling, np.where, shift and pct_change and you can't read those, the code is a black box you can't trust. Learn the basics first; then let the AI accelerate you, with you as the verifier.

Key takeaways

  • A library is a labelled toolbox; built-in ones (os, math) need no install, third-party ones (numpy, pandas) install once and import every session.
  • Working directory: os.getcwd() / os.chdir(); read_csv only finds files in the CWD; dir(lib) lists contents.
  • Five imports; alias form is preferred, import * discouraged, and the library name always comes first (from B import A).
  • Namespace overriding: the later import wins — stay explicit (math.sin) for traceable code.
  • Attribute vs method: df.shape (value, no parens) vs df.describe() (action, parens).
  • Brackets: round = call/tuple/args, square = list/index/slice, curly = set/dict/f-string.
  • Magic commands: %who, %whos, %pwd, %history, and %timeit — and remember work rarely scales linearly.
