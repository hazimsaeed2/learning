① Core Example — SVM Predicts Next-Day Open Direction (87 % / 80 %)

WHAT

Build three classifiers on Tesla daily data (≈1 500 rows). Logistic Regression & Decision Tree give ≈53 % / 50 %. Then the lecture changes two things at once: target from close-to-close to open-to-open and algorithm to SVM, yielding training 87 % and testing 80 %. The class discussion leans toward target/feature alignment as the main driver of the improvement.

WHY

Features are built mainly from close prices. At market close you have the full day's information (close, high, low, volume). The lecture's explanation is that predicting next day's open gives the model an extra same-day information window from open to close; open prices are still random, but are framed as less random than close prices in this discussion.

HOW — Full SVM Workflow

# --- 1. Import libraries ---
import numpy as np, pandas as pd                               # numerical + dataframes
from sklearn.svm import SVC                                     # Support Vector Classifier
from sklearn.preprocessing import StandardScaler                # feature scaling (required for SVM)
from sklearn.model_selection import train_test_split            # split data
from sklearn.metrics import confusion_matrix, classification_report  # evaluation

# --- 2. Read data ---
data = pd.read_csv('TSLA.csv', index_col='Date', parse_dates=True)  # Tesla OHLCV, 2014+

# --- 3. Create NEW target: next day open > today open ---
data['returns'] = np.log(data['Close']).diff()                  # log returns for features
data['target']  = np.where(data['Open'].shift(-1) > data['Open'], 1, 0)
#   shift(-1) = next row's open; > today open → 1, else 0

# --- 4. Feature extraction (same as MLT-01) ---
for p in [5,10,15,20]:                                          # volatility: rolling std
    data[f'std_{p}'] = data['Close'].rolling(p).std()
for p in [10,15,20,25,30]:                                     # trend: rolling mean
    data[f'ma_{p}']  = data['Close'].rolling(p).mean()
for p in [3,6,9]:                                               # momentum: cumulative pct returns
    data[f'mom_{p}'] = data['Close'].pct_change(p)
data['vol_ma4']      = data['Volume'].rolling(4).mean()         # volume trend
data['close_to_open'] = abs(data['Close'] - data['Open'])       # intraday movement

data.dropna(inplace=True)                                       # remove NaN from rolling

# --- 5. Split (shuffle=False for time series!) ---
features = [c for c in data.columns if c not in ['target','returns','Open','High','Low','Close','Volume']]
X = data[features];  y = data['target']
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.25, shuffle=False)

# --- 6. Scale features (fit on TRAIN only!) ---
scaler = StandardScaler()
X_train_s = scaler.fit_transform(X_train)                      # learn mean/std from train
X_test_s  = scaler.transform(X_test)                            # apply train's mean/std to test

# --- 7. Train SVM ---
model = SVC(kernel='rbf', random_state=42)                      # RBF = Radial Basis Function kernel
model.fit(X_train_s, y_train)

# --- 8. Evaluate ---
print("Train:", model.score(X_train_s, y_train))                # 0.87 (87%)
print("Test :", model.score(X_test_s,  y_test))                 # 0.80 (80%)
print(confusion_matrix(y_test, model.predict(X_test_s)))        # high diagonal numbers ✓
print(classification_report(y_test, model.predict(X_test_s)))   # precision ~80-83%
WORKED EXAMPLE
Train acc 87 % → bias? No (87 > 50).
Test acc 80 % → variance? 87−80 = 7 % gap, which sits inside Jay's usual 8-10 % thumb rule, so he treats the model as not materially overfitted here.
Precision class-1 ≈ 83 % → when model says "up," stock goes up 83 % of the time.
TERMINOLOGY
BTST = Buy Today Sell Tomorrow: buy at close, sell at next open.
EMH = Efficient Market Hypothesis: prices reflect all info.
RBF = Radial Basis Function kernel (SVM default).
GOTCHA
SVM requires scaling — skip it and features with larger values dominate the hyperplane. The lecture argues that the open-to-open setup improves because close-based features add extra same-day information at the market close. If features were based only on open prices, that advantage could shrink sharply.

② Model Fitting — Bias & Variance

WHAT

Why .fit() not .train()? Because the model tries to fit the shape of the data. Three states: underfitted (too simple, doesn't learn), well-fitted (replicates true data shape), overfitted (memorizes training data, wiggles beyond it). Two error sources explain these: bias and variance.

WHY

Underfitting and overfitting are both bad. Only well-fitted models generalize to unseen data. If you can't diagnose which error is present, you can't fix your model — you'll waste time changing the wrong thing (algorithm vs. features vs. data size).

BIAS (→ Underfitting)

Definition: Model's inability to learn the association between features and target, even after training multiple times.

Kid analogy: No matter how many times you show a kid apples, the kid keeps saying "orange."

Quantification: Training accuracy < 50 % → biased (worse than random chance for binary classification).

Causes: Algorithm too simple, insufficient data, non-predictive features.

VARIANCE (→ Overfitting)

Definition: Over-reliance on training data. Model memorizes training data instead of learning patterns.

Kid analogy: Kid memorizes one set of apples; replace with a different variety → can't recognize them.

Quantification: Training − Testing accuracy > 8–10 % → high variance (Jay's rule of thumb).

Causes: Model too complex (e.g., deep tree), too few training samples relative to features.

WORKED EXAMPLE — Bullseye Diagram
Low bias + Low variance → darts cluster on bullseye (well-fitted).
High bias + Low variance → darts cluster but OFF bullseye (consistently wrong).
Low bias + High variance → darts scattered AROUND bullseye.
High bias + High variance → darts scattered AND off-target.
TERMINOLOGY
Bias = systematic error; model predicts elsewhere.
Variance = sensitivity to training data changes.
Generalization = performs well on unseen data.
GOTCHA
Bias is checked on TRAINING accuracy only (not test). A model with training 70 % and test 49 % is NOT biased (70 > 50) — it has HIGH VARIANCE. Don't confuse low test accuracy with bias.

③ Confusion Matrix — The Foundation of Model Evaluation

WHAT

A 2×2 matrix showing how predictions map to actual classes. Named from the prediction side: "False Positive" = falsely predicted as positive. Class 1 = positive class (action / long position), Class 0 = negative class (no position). Want HIGH numbers on the diagonal (correct predictions).

WHY

Accuracy alone can't tell you WHERE the model fails. The confusion matrix reveals whether the model is predicting the actionable class (class 1) correctly — which is what traders care about. It drives precision and recall calculations.

HOW — Constructing the Matrix

# 1000 records: 400 actual class-0, 600 actual class-1
# Model predicts:
#   Of 400 class-0: 350 predicted 0 (correct), 50 predicted 1 (wrong)
#   Of 600 class-1: 100 predicted 0 (wrong), 500 predicted 1 (correct)

#                    Predicted 0      Predicted 1
#  Actual 0    ┌──────────────────┬──────────────────┐
#              │  TN = 350        │  FP = 50         │   ← True Neg / False Pos
#  Actual 1    ├──────────────────┼──────────────────┤
#              │  FN = 100        │  TP = 500        │   ← False Neg / True Pos
#              └──────────────────┴──────────────────┘

from sklearn.metrics import confusion_matrix
cm = confusion_matrix(y_test, y_pred)   # rows = actual, cols = predicted
# cm = [[TN, FP],
#        [FN, TP]]
WORKED EXAMPLE
TN=350, FP=50, FN=100, TP=500.
Accuracy = (350+500)/1000 = 85 %.
Diagonal sum = 850 correct out of 1000.
Off-diagonal = 150 errors.
TERMINOLOGY
TP = True Positive (correctly predicted positive).
FP = False Positive (falsely predicted as positive; Type I error).
FN = False Negative (falsely predicted as negative; Type II error).
TN = True Negative (correctly predicted negative).
GOTCHA
In trading, FP = model says "go long" but stock goes down (lose money). FN = model says "no trade" but stock goes up (miss opportunity). Which is worse depends on your strategy. sklearn's confusion_matrix() puts TN top-left, TP bottom-right.

④ Evaluation Metrics — Accuracy, Precision, Recall

WHAT

Three metrics to evaluate classifiers. Accuracy = overall correctness but misleading with imbalanced data. Precision = "when model says up, how often is it right?" Recall = "of all actual up days, how many did model catch?"

WHY

With imbalanced data (e.g., predict 5 %+ moves: 46 positive vs 1 454 negative), a model predicting ALL zeros gets 93 % accuracy while being useless. Precision and recall focus on the positive class — the actionable class in trading.

Accuracy

(TP + TN) / (TP + FP + TN + FN)

= correct predictions / total. First check, but can be misleading. E.g., 930/70 split → 93 % by predicting all 0s.

Precision

TP / (TP + FP)

"When model says stock goes up, 90 % of the time it does." High precision = high confidence in signals. 500/(500+50)=90.9 %.

Recall (Sensitivity)

TP / (TP + FN)

"Of all actual up days, model catches 83 %." High recall = few missed opportunities. 500/(500+100)=83.3 %.

from sklearn.metrics import classification_report, accuracy_score

print(accuracy_score(y_test, y_pred))                 # single number: 0.85
print(classification_report(y_test, y_pred))          # precision, recall per class
#               precision    recall  f1-score   support
#            0       0.78      0.88      0.83       160
#            1       0.83      0.80      0.82       210
#     accuracy                           0.80       370
WORKED EXAMPLE — Imbalanced Trap
Target: stock rises >5 %. Data: 1 454 zeros, 46 ones.
Lazy model predicts 0 always → accuracy 96.8 %!
Precision = 0/0 = undefined. Recall = 0/46 = 0 %.
Precision & recall expose the fraud.
TERMINOLOGY
Imbalanced data = class distribution heavily skewed.
F1-score = harmonic mean of precision & recall.
Support = number of actual instances per class in test set.
GOTCHA
Also check test-set balance after splitting! Original data balanced doesn't guarantee test set is. If test set is imbalanced → accuracy is misleading even if training set was fine. Jay prefers precision/recall over resampling techniques (SMOTE etc.).

⑤ Decision Tree — Splitting Data by Conditions

WHAT

Algorithm that breaks the dataset into subsets where each subset ideally has records of a single class. Builds an inverted tree: root node → branches (conditions) → decision nodes → leaves (predictions). Conditions are found using Gini index (default) or Entropy.

WHY

Key advantage: NO FEATURE SCALING NEEDED — works on raw data. Super fast to train and predict (critical for production with millions of rows). Interpretable — you can visualize the tree and see exact rules. Gini is go-to for financial data; entropy also works.

from sklearn.tree import DecisionTreeClassifier, plot_tree
import matplotlib.pyplot as plt

# --- Define and train ---
dt = DecisionTreeClassifier(criterion='gini',       # 'gini' or 'entropy'
                            max_depth=3,             # restrict tree height to 3 levels
                            random_state=42)
dt.fit(X_train, y_train)                             # NO scaling needed!

# --- Evaluate ---
print("Train:", dt.score(X_train, y_train))          # 57%
print("Test :", dt.score(X_test,  y_test))           # 50%  (7% diff — borderline variance)

# --- Visualize the tree ---
plt.figure(figsize=(20,8))
plot_tree(dt, feature_names=features, filled=True, rounded=True)
plt.show()

# --- Feature importance ---
importance = pd.Series(dt.feature_importances_, index=features)
importance[importance > 0].sort_values().plot.barh() # only 5 of 15+ features used!
WORKED EXAMPLE
depth=3: uses only 5 features (ma_15, ma_25, std_5, vol_ma4, close_to_open). Misses momentum, other MAs.
depth=6: uses all features BUT train 70 % / test 49 % → HIGH VARIANCE.
Increasing depth = more features but more overfitting.
TERMINOLOGY
Root node = first split (entire dataset).
Decision node = intermediate condition.
Leaf = terminal node with class prediction.
Impure = mixed classes; Pure = single class.
Gini index = measure of impurity (0 = pure).
GOTCHA
Decision trees are PRONE TO OVERFITTING. Deeper tree = memorization. Always restrict max_depth. With depth=3, few features used; with depth=6, all features but 70 %/49 % → overfitted. This is the fundamental dilemma that leads to Random Forest.

⑥ Random Forest — Many Trees, Majority Vote

WHAT

Instead of one deep tree, train 100+ shallow trees. Each tree splits data differently, using different features. For prediction, all trees vote — majority class wins. This solves decision tree's dilemma: all features get utilized without increasing individual tree depth.

WHY

Single decision tree: restrict depth → few features; increase depth → overfitting. Random Forest: keep depth restricted per tree, but train many trees → all features utilized across the forest. Also: no scaling needed, works well with LARGE datasets (millions/billions of rows).

from sklearn.ensemble import RandomForestClassifier

# --- Define: 100 trees, each max depth 3 ---
rf = RandomForestClassifier(n_estimators=100,        # number of trees
                            max_depth=3,             # each tree limited to 3 levels
                            criterion='gini',        # splitting criterion
                            random_state=42)
rf.fit(X_train, y_train)                             # NO scaling needed!

print("Train:", rf.score(X_train, y_train))          # ~64-70%
print("Test :", rf.score(X_test,  y_test))           # ~50%  (still overfitted on THIS data)

# Result: even 100 trees couldn't fix the close-to-close target setup here
# → Lecture takeaway: the issue appears more related to data/target design than to algorithm choice alone
WORKED EXAMPLE
100 trees, depth 3, Gini: train 70 % / test 50 %.
Switch to entropy: train 64 % / test 47 % (worse!).
Reduce to 50 trees: no improvement.
Three algorithms (LR, DT, RF) all ≈50 % test → in this example, the bottleneck appears to be the data/target setup more than the algorithm swap alone.
TERMINOLOGY
n_estimators = number of trees in the forest.
Ensemble = combining multiple models.
Majority voting = 51 of 100 trees say "up" → predict up.
Bagging = each tree trained on random subset of data.
GOTCHA
Random forests are STILL prone to overfitting — just less so than a single deep tree. Can't inspect which features overlap between trees (package limitation). Tree-based algorithms (DT & RF) are naturally more overfit-prone than regression-based models.

⑦ Support Vector Machine (SVM) — Hyperplanes & Kernel Tricks

WHAT

Finds a hyperplane (decision boundary) that maximizes the margin between classes. The nearest data points to the boundary are called support vectors. For non-linearly separable data, applies kernel tricks — transforms data into higher dimensions where it becomes linearly separable (e.g., 2D circles → 3D separable planes).

WHY

Shines with medium/small datasets (thousands of rows) and high-dimensional data (many features, few observations). Unlike tree-based models which need large data, SVM excels where you have limited data but rich features — common in quant trading. Requires scaling but handles complex non-linear boundaries via kernels.

from sklearn.svm import SVC                          # SVC = classification
# from sklearn.svm import SVR                        # SVR = regression variant

# --- Available kernels ---
# 'rbf'     = Radial Basis Function (default, most popular)
# 'linear'  = always linear boundary (no kernel trick)
# 'poly'    = polynomial boundary
# 'sigmoid' = sigmoid-shaped boundary

model = SVC(kernel='rbf', random_state=42)
model.fit(X_train_scaled, y_train)                   # MUST scale features first!

# --- Kernel trick intuition ---
# Problem: 2D data in concentric circles (inner=class A, outer=class B)
#   → No straight line can separate them
# Solution: SVM projects data into 3D → now a flat plane separates them
# The math (kernel function) does this WITHOUT actually computing 3D coords
WORKED EXAMPLE
Same Tesla data, same features, new target (open-to-open):
SVM + RBF kernel → train 87 %, test 80 %.
Confusion matrix: high diagonal (correct predictions).
Precision class-1 ≈ 83 %. Recall class-1 ≈ 80 %.
TERMINOLOGY
Hyperplane = decision boundary (line in 2D, plane in 3D, hyperplane in nD).
Margin = distance from boundary to nearest points.
Support vectors = nearest data points defining margin.
Kernel = function that transforms data for separation.
GOTCHA
SVM REQUIRES feature scaling (unlike trees). Without scaling, features with larger numerical ranges dominate. Not scalable to massive datasets (millions of rows) — use Random Forest instead. SVM model is a "black box" — harder to interpret than decision trees.

⑧ The Critical Insight — Target & Feature Design Often Matter More Than Algorithm Swaps

WHAT

Three different algorithms (Logistic Regression, Decision Tree, Random Forest) with close-to-close target all gave ≈50–53 % test accuracy. Later, the lecture changes both the target variable and the algorithm, and the SVM example reaches 80 % test accuracy. The takeaway is that target and feature design can matter at least as much as, and often more than, just swapping algorithms.

WHY

The lecture treats close-to-close prediction as much noisier because both features and target are tied closely to close prices. With an open-to-open target plus close-based features, the model now sees an extra intra-day information block before prediction time. Your "natural intelligence" in designing targets and features is presented as a major differentiator.

Algorithm Comparison (same features, close-to-close target)

Random Forest (100)70 %50 %Overfitted

After changing target (open-to-open)

AlgorithmTrainTestVerdict

Two changes happened together: algorithm + target. The class discussion leans toward target/feature alignment as the bigger reason for the improvement, but this exact demo does not isolate the two effects experimentally.

What To Try When Models Fail

  1. Increase dataset size (more history)
  2. Add more predictive features (ratio-based, e.g., price/MA)
  3. Design more logical target variable
  4. Change algorithm (least likely to help alone)

Combination of 1–3 gives highest chance of improvement.

WORKED EXAMPLE
Carlos's insight: "price / MA ratio" is more predictive than raw MA level. Raw MA_15 = 175 is meaningless across stocks. But price/MA_15 = 1.05 means "5 % above 15-day MA" — universally meaningful and captures overbought/oversold. Design features with financial logic, not just rolling statistics.
TERMINOLOGY
Predictive power = features' ability to predict target.
Feature engineering = creating informative features from raw data.
Information leakage = using future data in features (NOT the case with shift(-1) target creation — just labeling, not training).
GOTCHA
Don't keep changing algorithms hoping for better results — if 3 algorithms give similarly weak results, the lecture says to inspect data, features, and target design first. In this example, the open-to-open setup benefits from extra same-day information in close-based features; if features were open-only, that advantage could narrow.

KEY TAKEAWAYS

1. Evaluate Beyond Accuracy
Accuracy is misleading with imbalanced data. Use confusion matrix, precision (confidence in positive predictions), and recall (coverage of actual positives). In trading, we care most about precision — when model says "go long," is it right?
2. Diagnose Bias vs. Variance
Training acc < 50 % → bias (model too simple or bad features). Train − Test gap > 8–10 % → variance (overfitting). Decision trees overfit easily — restrict depth or use Random Forest. But even RF can overfit if data/target is poor.
3. Target & Features > Algorithm
Three algorithms with the earlier setup stay near 50 %. When the lecture changes both target and algorithm, the result jumps to 80 %. The main lesson is to design targets and features so the model has genuine predictive information, instead of relying on algorithm swaps alone.
4. Know Each Algorithm's Sweet Spot
Decision Trees: no scaling, interpretable, but overfit-prone. Random Forest: many trees, all features, large data. SVM: kernel tricks, medium data, needs scaling. Choose based on data size, feature count, and production constraints.
