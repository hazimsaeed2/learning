Backtesting using Blueshift

Week 22-2 · TBP-03 — why a serious backtester is event-driven, and how Blueshift turns that idea into working strategies.

There is a moment in every quant's life when a spreadsheet backtest that looked brilliant collapses in live trading. Almost always the culprit is the same: the backtest quietly let the strategy see the future, or it was written in a way that could never actually be deployed. This lecture is about the kind of software that prevents both failures — an event-driven engine — and about Blueshift, QuantInsti's cloud platform built around exactly that idea.

Where Blueshift lives

Trading happens across a vast range of speeds. At one extreme sits high-frequency trading, where decisions are made in less than a microsecond and the edge comes from cables, co-location and custom hardware. At the other extreme sits long-horizon investing, where you might rebalance monthly or quarterly. Blueshift is built for the low-frequency end of that spectrum — anything from tens of milliseconds out to quarterly. The reason is philosophical as much as technical: in low-frequency trading your advantage comes from the idea, not from the wires. Blueshift is therefore optimised for research and reasoning, not for shaving nanoseconds.

What makes an engine worth trusting

The lecturer lists five things any serious algorithmic engine must have, and they are worth internalising because they explain every design choice that follows.

First, it must be event-driven. This single property protects you from look-ahead bias automatically, and — just as importantly — it lets the same code that backtests your strategy also trade it live. Second, it must run a complex-event-processing engine, which is a fancy way of saying it is stateful: it remembers what has happened, so your strategy can react to accumulated state, not merely to the latest tick. A naive "if this, then that" rule engine is stateless and cannot do this. Third, it needs high-quality data — free of survivorship bias, free of look-ahead bias, and crucially point-in-time. Fourth, it needs institutional-grade APIs that are complete, unambiguous and fast. And fifth, it must be sound software engineering.

Roughly 95% of trading code is plumbing — fetching data, cleaning it, managing orders, reconnecting when the line drops, reconciling positions. Only about 5% is the actual strategy. Blueshift owns the 95% so that you can spend your time on the 5% that is genuinely yours.

Vectorised versus event-driven — the heart of it

A vectorised backtest loads the entire price history into memory and computes everything in one sweep: signals across the whole series, then the resulting profit and loss. It is gloriously fast, and for a first sniff-test of an idea it is perfectly fine. But it has a dangerous flaw. Because the whole past and the whole future sit together in one dataframe, a single careless line — a rolling average that includes today's close, a shift in the wrong direction — quietly leaks tomorrow's information into today's decision. That is look-ahead bias, and it is the most common way a backtest lies to you.

In the companion notebook we make this concrete on ten years of real Reliance data. The same one-line signal — take a position equal to the sign of the day's return — is run two ways. When it is allowed to peek at that day's return before deciding, one rupee grows to roughly ten trillion rupees, a factor of about 10¹³; when the identical signal is shifted by a single bar so that it uses only information available before the decision, it quietly loses about 5% — all while the stock itself rose 537%. The entire gap is fantasy manufactured by look-ahead bias.

An event-driven engine makes that fantasy impossible. It processes the world the way the world actually arrives: a bar comes in, the strategy reacts, it perhaps places an order, it updates its internal state, and then it waits for the next bar. It is slower, but it can only ever see the past, so look-ahead bias cannot occur by construction. It handles stateful rules naturally, and — the decisive advantage — the backtest code is the live-trading code. Blueshift's event engine is powered by Apache Arrow, and the practical advice is simple: screen ideas quickly with vectorised code, then move to the event-driven engine the moment you are serious.

Point-in-time data

Of all the data sins, the subtlest is failing to be point-in-time. When your backtest looks at a price on some past date, it must see that price as it genuinely appeared on that date — with only the corporate actions and futures-roll adjustments that had happened up to that point applied, and nothing from the future. A charting website that has silently back-adjusted a high-dividend stock will show you a history that never existed. In the notebook we take Reliance a hundred trading days before its 2 July 2020 dividend: the point-in-time price, as it genuinely appeared that day, is ₹652.10, while a naively back-adjusted series — one that has leaked the future dividend back into the past — reports ₹635.17. That ₹16.93 gap is a future event smuggled into the past.

The three pillars

Every institutional platform, Blueshift included, rests on three pillars, and once you see them the code stops looking mysterious.

The first pillar is callbacks — functions you register with the engine, saying in effect "when this kind of event happens, call me." The lifecycle callbacks are initialize (run exactly once at the very start; without it the strategy is invalid), before_trading_start (run each day about thirty minutes before the open), and handle_data (run on every bar, typically every minute). On top of these you can register your own with schedule_function, attaching date rules and time rules — start of month, every day, thirty minutes after the open, and so on. There is even a schedule_soon that runs as soon as possible without blocking the event manager.

The second pillar is the context — an object that carries the algorithm's state. The elegant trick the lecturer highlights is that because context is an ordinary Python object passed to every callback, you can simply attach your own attributes to it: context.params, context.signals, context.securities. This gives you shared variables across functions without resorting to global variables, which everyone agrees are a poor pattern.

The third pillar is the set of domain APIs — symbol() to turn a ticker into a rich asset object, the order functions, the data portal, set_stop_loss, and so on.

Orders, and why targeting beats absolute

Blueshift offers two families of order. Absolute orders (order, order_value, order_percent) say "buy five units." Targeting orders (order_target and friends) say "have a position of five units." The difference matters more than it first appears. A targeting order trades only the difference between where you are and where you want to be: if you hold three and target five, it buys two; if you hold ten and target five, it sells five. The lecturer's vivid warning is about "machine-gunning": if you write an absolute order inside a condition like RSI > 70, the order fires on every bar the condition holds, piling up an unintended position. A targeting order solves this automatically — once the target is reached it does nothing. Every order returns an id (or a list of ids if it took retries); a return of None means the order failed, which is a signal you must handle.

The data portal

When a strategy asks for data it does not receive a dataframe — it receives a data portal, an object with two methods: current for the latest value and history for a window of bars. Blueshift always returns the simplest possible shape. Ask current for one asset and one field and you get a bare float; ask for several assets or several fields and you get a Series; ask for many of both and you get a DataFrame. history simply goes one dimension higher — float becomes Series, Series becomes DataFrame, DataFrame becomes a multi-index frame. And every history query is strictly point-in-time, returning only bars up to the present moment of the simulation.

From lifecycle to strategy: three live walkthroughs

With the machinery understood, the lecturer walks through three strategies live in class, each illustrating a different layer of the platform. Read these as demonstrations of the API, not as signals — none comes with a profit-and-loss claim, and their job is to make the moving parts visible.

The 9:20 short-straddle

The first is a pure options-income idea. At 9:20 in the morning you sell an at-the-money call and an at-the-money put, collecting the premium of both. Each leg is given its own forty-percent stop-loss and take-profit, and whatever survives is squared off at 15:00. The economics are all about theta: as the day passes, the options decay in your favour, but a sharp move in the underlying hurts whichever leg it runs against. The instructor walks this one through live to demonstrate scheduling — enter at 9:20, exit at 15:00 — and leg-wise stops; no profit figure is claimed. The shape is the lesson: quiet days quietly keep their premium, while trending days have their losses capped — one leg stops out while the other take-profits. The reason such a book can net positive over time is the variance-risk premium, the tendency for the implied volatility you sell to exceed the volatility that actually materialises.

Bollinger breakout with an EMA fallback

The second strategy is the flagship walkthrough and shows the full lifecycle in code. It computes Bollinger bands and then a "distance to upper" measure, (upper − last)/(upper − lower) × 100. When the price hugs the upper band the measure is near zero and the strategy buys the anticipated breakout; when it hugs the lower band the measure is near one hundred and it sells; and when the price sits in the middle the strategy falls back on a simple fast-versus-slow EMA crossover. The beautiful part is structural: all of this lives inside a single signal() function, so swapping Bollinger for RSI or a Fibonacci rule means editing one function and nothing else. This is the flagship walkthrough, shown to make the full lifecycle visible in code rather than to advertise a return. A final detail the lecturer stresses is that the backtest respects your capital: run the same Nifty-and-Bank-Nifty-futures strategy with only ₹50,000 and not a single order is placed, because one lot already exceeds the account — the equity curve is a flat line.

The pipeline API

The first two strategies hand-listed their universe. But when the universe is large, or defined by a rule rather than a list, you reach for the pipeline API. A pipeline is a lazy computation graph — a directed acyclic graph in the same family as TensorFlow's — built from factors (each producing a number per asset) and filters (each producing a boolean). The canonical example is time-series momentum: a liquidity filter keeps only heavily-traded names, a positivity filter keeps only those with positive twelve-month returns, a return factor ranks the survivors, and the top five are equal-weighted at twenty percent each. Nothing is computed when you define the graph; the engine evaluates it lazily, only when you call pipeline_output during the strategy.

The three shipped strategies, on real data

Beyond the live walkthroughs, the lecture ships three complete, downloadable strategy files — and these are the ones worth running for yourself. In the companion notebook each is reproduced offline through a small event-driven engine that mirrors Blueshift's own initialize / handle_data / order_target_percent interface, then backtested on real NSE daily data from 2015 to 2024 with no transaction costs.

The first, MACD.py, is a textbook trend-follower on Tata Steel: go long when the MACD line is above its signal line, short when it is below. On real data it loses 72.8% over the decade, against a buy-and-hold that made 436%. The reason is instructive — it whipsaws on every crossover and, fatally, spends nearly half its life short a stock in a long structural uptrend. The second, rsi_strategy.py, buys Tata Steel and Reliance when their fourteen-day RSI falls below thirty and sells above seventy. Because those extremes are rare, it sits in cash roughly four days in five and ekes out just +5.1% across ten years. The third, Statarb using PCA.py, trades the Ambuja-Cement-versus-ACC pair, using the first principal component of the two price series as a hedge ratio and trading the z-scored spread. The two stocks correlate 0.93, which looks like a perfect pair — yet the naive spread trade loses 78.5%, because a high correlation is not the same thing as a stable, mean-reverting cointegration, and the spread simply drifts away.

None of this is a failure of the platform; it is the whole pedagogical point. These strategies exist to teach the architecture and the API, and their honest, disappointing returns are the lecture's own message made concrete: a backtest that passes is the beginning of research, not proof of an edge.

Letting an LLM write the code — carefully

The lecture is refreshingly modern about large language models. You can absolutely have ChatGPT or Copilot generate Blueshift strategies, and the Blueshift Developer Kit ships LLM guides on GitHub that dramatically reduce invented API calls and often produce working code in one shot. But there is a hard limit the lecturer keeps returning to: you can never fully test a trading strategy, because you cannot simulate every market state the way you can write a complete test suite for a website. So the human's job shifts.

Writing can be done by the LLM. Your job is to understand what it produced — to focus on the why, not the how.

The honest closing note

A passing backtest is necessary but never sufficient. The disciplined path runs further: forward-test the idea on paper or with small capital, watch how it behaves on data it has never seen, and ultimately build a portfolio of strategies rather than betting everything on one. Blueshift's whole design — event-driven, stateful, point-in-time, the same code in research and in production — exists to make that disciplined path the path of least resistance.

The one-paragraph version. Blueshift is an event-driven, low-frequency backtesting and trading platform. Event-driven beats vectorised because it cannot leak the future and the same code runs live. It rests on three pillars — callbacks you register, a context object that carries state, and domain APIs. Prefer targeting orders so you trade only the difference; trust the point-in-time data portal that returns the simplest shape; map the quant lifecycle (universe → signal → target → rebalance → review) directly onto your functions; reach for the pipeline factor/filter graph when the universe is large; and remember that a backtest is the beginning of the work, not the end.
