HOW: Data Types & Date Initialization

WHAT: Python distinguishes between tuples, lists, strings, and datetime objects. You must explicitly create datetime objects for date-based functions.

# Tuple (round brackets) — immutable, ordered
start_date = (2017, 1, 1)          # type: tuple — yf CANNOT use this!

# String (quotes) — yf CAN use this
start_date = "2017-01-01"          # type: str — yf accepts this format

# Datetime (explicit) — most reliable
start_date = dt.date(2017, 1, 1)   # type: datetime.date — yf accepts this
end_date   = dt.date(2020, 1, 1)   # year=2020, month=1, day=1

⚠️ WARNING — Tuple dates crash yf.download(): If you pass a tuple like (2017, 1, 1) to yf.download(), you get: AttributeError: 'tuple' object has no attribute 'timestamp'. Always use a string "2017-01-01" or dt.date() object.

5️⃣ ARRAYS VS. LISTS — Why NumPy Arrays Enable Fast Math

WHAT: Lists Can't Do Element-wise Math

WHY: Lists repeat on *; arrays multiply each element. This matters because strategy signals require element-wise operations on thousands of rows simultaneously.

number_list = [1, 2, 11, 22, 5, 6, 8]   # Python list

number_list * 3                          # Repeats the list 3 times!
# Output: [1, 2, 11, 22, 5, 6, 8, 1, 2, 11, 22, 5, 6, 8,
#          1, 2, 11, 22, 5, 6, 8]

number_array = np.array(number_list)     # Convert list → NumPy array

number_array * 3                         # Multiplies EACH element by 3
# Output: array([ 3,  6, 33, 66, 15, 18, 24])

WORKED EXAMPLE with Core Data (first 5 Close prices):

Close PriceList × 2Array × 2
100100 (just repeats)200
102102 (just repeats)204
9898 (just repeats)196
105105 (just repeats)210
103103 (just repeats)206

List × 2 gives [100,102,98,105,103,100,102,98,105,103] — doubled in length, not value!

HOW: The For-Loop Alternative (Slower)

WHAT: Without arrays, you must iterate with a for loop — one element at a time.

# The SLOW way — iterate one by one
for value in number_list:        # Loop through each element
    print(value * 3)             # Multiply one at a time
# Output (one per line): 3, 6, 33, 66, 15, 18, 24

# The FAST way — vectorized (all at once)
result = np.array(number_list) * 3  # Done concurrently
# Output: array([ 3,  6, 33, 66, 15, 18, 24])

KEY INSIGHT — Vectorized vs. Event-Driven:

• Vectorized = all elements processed concurrently (fast, no loop)

• Event-driven = elements processed one by one via for-loop (slow, sequential)

• This same distinction applies to backtesting strategies — see Section 8.

6️⃣ MOVING AVERAGE CROSSOVER STRATEGY — The Full Workflow

STEP 1: Calculate Indicators

WHAT: .rolling(N).mean() computes the Simple Moving Average (SMA) — the arithmetic mean of the last N closing prices.

WHY: The crossover of short SMA and long LMA generates buy/sell signals.

short_window = 21               # Short lookback
long_window  = 63               # Long lookback

# SMA: average of last 21 Close prices
data['SMA'] = data['Close'].rolling(
    short_window).mean()

# LMA: average of last 63 Close prices
data['LMA'] = data['Close'].rolling(
    long_window).mean()

STEP 2: Shift & Generate Signals

WHAT: .shift(1) moves all values down by 1 row — giving "yesterday's" value in today's row.

# Get yesterday's SMA & LMA
data['SMA_prev'] = data['SMA'].shift(1)
data['LMA_prev'] = data['LMA'].shift(1)

# Drop rows with NaN (first 63 rows)
data.dropna(inplace=True)
# 962 rows → 899 rows

Signal generation with np.where:

# LONG signal: SMA crosses ABOVE LMA
data['signal'] = np.where(
  (data['SMA'] > data['LMA']) &
  (data['SMA_prev'] < data['LMA_prev']),
  1, 0)                    # 1=BUY, else 0

# SHORT signal: SMA crosses BELOW LMA
data['signal'] = np.where(
  (data['SMA'] < data['LMA']) &
  (data['SMA_prev'] > data['LMA_prev']),
  -1, data['signal'])      # -1=SELL, else keep

data['signal'].value_counts()
# 0     891 (no signal)
# 1       4 (4 BUY crossovers)
# -1      4 (4 SELL crossovers)

# Replace 0 → NaN, then forward-fill
data['signal'].replace(

data['signal'].fillna(

Squaring Off (End of Backtest)

  # Force-close any open position on the last day
  # Exit signal must be the OPPOSITE of the final open position
  exit_signal = -data['signal'].iloc[-1]
  data.iloc[-1, data.columns.get_loc(

  # If final position was +1 (long), exit_signal becomes -1

In the lecture's worked example, the final open trade happened to be long, so the square-off signal was -1. That is an example, not a universal rule.

⚠️ WARNING — inplace=True: dropna(), replace(), fillna() do NOT modify the DataFrame unless you pass inplace=True. Without it, the original stays unchanged.

7️⃣ RETURNS — Simple vs. Log & Strategy Evaluation

WHAT: Simple Returns vs. Log Returns

Simple Return = (Today − Yesterday) / Yesterday = .pct_change()

Log Return = ln(Today / Yesterday) = np.log(Close / Close.shift(1))

# Buy-and-hold return (simple daily % change)
data['buy_hold_returns'] = data['Close'].pct_change()

# Strategy return = position × buy-and-hold shifted by 2
data['strategy_returns'] = (
    data['signal'] *                  # Your position (+1 or -1)
    data['buy_hold_returns'].shift(2) # Return shifted by 2 days
)

# Log returns (alternative):
data['log_returns'] = np.log(
    data['Close'] / data['Close'].shift(1))

WORKED EXAMPLE (using Core Data):

DayCloseSimple ReturnLog Return
1100NaNNaN
2102+2.00%+1.98%
398−3.92%−4.00%
4105+7.14%+6.90%
5103−1.90%−1.92%

Day 2 Simple: (102 − 100) / 100 = 2/100 = 0.0200 = +2.00%

Day 2 Log: ln(102/100) = ln(1.02) = 0.0198 = +1.98%

Day 3 Simple: (98 − 102) / 102 = −4/102 = −0.0392 = −3.92%

Day 3 Log: ln(98/102) = ln(0.9608) = −0.0400 = −4.00%

For small daily moves, simple ≈ log. For large moves, they diverge.

WHY shift(2)? — Avoiding Look-ahead Bias

WHAT: You shift returns by 2 because: shift(1) = signal appears next day, shift(2) = return measured the day after.

Day 4 (Jan 5): Signal generated at close = 105

Day 5 (Jan 6): You enter the trade at close = 103

Day 6 (Jan 9): close = 107 → Return = (107 − 103)/103 = +3.88%

Signal Day 4 → Enter Day 5 → Return Day 6 = 2 shifts

Compounding Returns (Equity Curve)

# Simple returns: compound with cumprod
data['buy_hold_equity'] = (
    1 + data['buy_hold_returns']  # Add 1 to each return
).cumprod()                       # Cumulative product

data['strategy_equity'] = (
    1 + data['strategy_returns']
).cumprod()

# Log returns: just add (cumsum)!
data['log_equity'] = data['log_returns'].cumsum()

# Total return percentage
total_bh = (data['buy_hold_equity'].iloc[-1] - 1) * 100
total_st = (data['strategy_equity'].iloc[-1] - 1) * 100

WORKED EXAMPLE (compounding):

Day 1: equity = 1.0000

Day 2: 1.0000 × (1 + 0.0200) = 1.0000 × 1.02 = 1.0200

Day 3: 1.0200 × (1 − 0.0392) = 1.0200 × 0.9608 = 0.9800

Day 4: 0.9800 × (1 + 0.0714) = 0.9800 × 1.0714 = 1.0500

📊 VISUAL: Equity Curve — Strategy vs Buy-and-Hold

The strategy outperforms because it goes flat (or short) during market declines, avoiding drawdowns. The green line is smoother because losses are cut when SMA crosses below LMA.

💡 WHEN TO USE WHICH:

• Simple returns = short-term, human-readable ("rose 5% today")

• Log returns = statistical models, time-additive (just add them)

• Log is cheaper: cumsum (addition) vs. cumprod (multiplication)

8️⃣ VECTORIZED vs. EVENT-DRIVEN BACKTESTING — Two Approaches Compared

VECTORIZED (Fast, All-at-once)

WHAT: Apply conditions to the entire DataFrame at once using np.where(). No loop.

WHY: ~100× faster than event-driven. Good for prototyping indicator-based strategies.

# ALL 899 rows checked simultaneously
data['signal'] = np.where(
    (data['SMA'] > data['LMA']) &
    (data['SMA_prev'] < data['LMA_prev']),
    1,                          # Where True → 1
    0                           # Where False → 0
)
# No for-loop. One operation. Done.

WORKED EXAMPLE:

RowSMA>LMA?prev SMA<LMA?Signal
100NoNo0
101NoNo0
102Yes ✓Yes ✓1 (BUY)
103YesNo0

All 899 rows evaluated in one step — no iteration.

EVENT-DRIVEN (Slow, Row-by-row)

WHAT: Loop through each row with for, checking conditions one at a time.

WHY: Required for live trading, tick-by-tick strategies, and handling slippage/transaction costs dynamically.

trade_price = []                  # Entry prices
position = []                     # Positions (+1/-1)
trade_date = []                   # Trade dates

for i in range(len(df)):          # Each row, one by one
    if (df['SMA'].iloc[i] > df['LMA'].iloc[i] and
        df['SMA'].iloc[i-1] < df['LMA'].iloc[i-1]):
        trade_price.append(df['Close'].iloc[i])
        position.append(1)        # BUY
        trade_date.append(df.index[i])
    elif (df['SMA'].iloc[i] < df['LMA'].iloc[i] and
          df['SMA'].iloc[i-1] > df['LMA'].iloc[i-1]):
        trade_price.append(df['Close'].iloc[i])
        position.append(-1)       # SELL
        trade_date.append(df.index[i])
# Same 8 signals — checked ONE ROW AT A TIME

🎯 KEY TAKEAWAYS

1. Always use multi_level_index=False in yf.download() — default yfinance data is already adjusted for corporate actions (auto_adjust=True).

2. .shift(2) is critical: shift(1) for signal delay + shift(1) for return calculation = 2 total shifts to avoid look-ahead bias in strategy returns.

3. Simple returns compound with (1+r).cumprod(); log returns just add with .cumsum() — log is computationally cheaper and time-additive.

4. Vectorized (np.where) is fast for prototyping; event-driven (for loop) is needed for live trading with slippage and transaction costs (DMP-03/04).

THE MASTER FORMULA: Data → Indicators (.rolling().mean()) → Signals (np.where crossover) → Positions (forward-fill) → Returns (.shift(2) × position) → Equity (.cumprod()) → Evaluate

THE ONE THING TO NEVER FORGET: Never calculate strategy returns without shifting by 2 — otherwise you're using tomorrow's prices today (look-ahead bias), making your backtest results meaningless.
