Teaching Your Strategy to Trade by Itself

Week 10-2 · Trading & Backtesting Platforms · TBP-01 (with Jay)

Everything so far has been preparation. You learned to dream up a strategy, to quantify it, and to validate it against history with vectorized and event-driven backtests. This lesson crosses the bridge from research to action: how to take a validated idea and plug it into a real broker so that a Python program — not your clicking finger — fetches data, places orders, and manages positions. The vehicle is the Interactive Brokers Python API.

1. The shape of the journey

It helps to see where automation sits: idea → quantify → validate → automate → connect to broker → go live. "Automation" sounds grand, but concretely it just means a list of ordinary trading operations performed by code: connecting to the broker, fetching historical data, placing and modifying and cancelling orders, checking order status and positions, and streaming live quotes. Which of these you can automate depends entirely on what your broker's API exposes.

Do it by hand first. The instructor is emphatic: before you automate an operation, perform it manually in the broker's terminal. Once you've placed an order by hand and watched it appear in the activity pane, you understand exactly what your code is asking the broker to do — and you'll debug far faster.

2. Meeting the terminal: TWS

Interactive Brokers (IB) is a broker serving around 35 countries, which is why students from everywhere can follow along. Its desktop application is TWS — the Trader Workstation — a cross-platform, point-and-click terminal you install and log into. Two login modes matter: live trading and paper trading. Paper trading is simulated, and it's the perfect sandbox: you receive the same live market data as a funded account, but the fills are simulated, so you can rehearse a strategy with zero risk.

Inside, the window is a mosaic of panes — a monitor/watchlist, news, an order-entry box, a chart, an info pane, and an activity pane that logs your orders and trades. IB calls each tradable instrument a contract. A neat detail: a continuous future automatically rolls to the next contract at expiry, so your watchlist never silently goes stale. Place a buy order on a weekend and it simply sits at the broker until the exchange opens — which also means you can still cancel it, because it never reached the market.

3. What an API really is

Strip away the mystique and an API (Application Programming Interface) is just a piece of software that sits between two parties and lets them talk in a structured, programmatic way. One party is the client, who makes requests; the other is the server, who responds. The class offered lovely analogies — a waiter carrying your order to the kitchen and food back, or simply a connector between two applications.

For us the roles are fixed: the client is our Python program, and the server is the broker (TWS). IB's API arrives as a Python package called ibapi. (It's worth knowing other shapes exist: some APIs are sets of URLs you call over HTTP — the subject of the next class — and FIX is the heavyweight standard that financial institutions use to talk to each other.)

4. Two ways to converse: synchronous and asynchronous

Communication can be synchronous — the client sends one request and then sits idle until the answer comes back, handling one thing at a time — or asynchronous, where the client fires off many independent requests and receives the answers whenever they're ready, possibly out of order or in pieces, all while doing other work. Streaming live prices is impossible without the asynchronous style.

The fact that shapes everything: the IB API is asynchronous only. You don't get to choose — the API is built that way, and that single design decision is the reason a background listener and threads show up in every IB program. In the notebook a quick simulation made the difference tangible: three requests handled synchronously took about 0.15 seconds of idle waiting, while the asynchronous version overlapped them and finished in about 0.06 seconds.

5. The heart of ibapi: EClient and EWrapper

A Python package is just a bundle of classes, and in ibapi two of them carry the whole story. The EClient class holds the methods you call to make requests — connect, fetch data, place orders. The EWrapper class holds the callback methods through which the server's responses arrive. Two helper classes round things out: Contract (a financial instrument) and Order (an order to place).

They come in matched pairs. Every request method has one or more callbacks that deliver its answer:

You call (EClient request)The server calls back (EWrapper)
reqCurrentTime()currentTime(time)
reqContractDetails()contractDetails(...)
reqHistoricalData()historicalData(...)

The pattern you'll write a hundred times is to fuse both sides through multiple inheritance into a single custom class, conventionally called Strategy, then create one object named app:

class Strategy(EClient, EWrapper):
    def __init__(self):
        EClient.__init__(self, self)   # the 'double self' initialises both parents
    def currentTime(self, t):          # a callback — the SERVER calls this, not you
        print(datetime.fromtimestamp(t))

app = Strategy()
app.connect("127.0.0.1", 7496, clientId=1)
app.reqCurrentTime()                   # request out -> currentTime() fires

That curious EClient.__init__(self, self) — passing self twice — is the one line you simply copy: it wires the wrapper into the client so both parents initialise together. The three arguments to connect are worth memorising: host is the IP address of the machine running TWS (127.0.0.1/localhost if it's the same computer), port is the number you set in the IB API settings (commonly 7496), and clientId is any integer you like — you can connect up to 32 clients to one TWS, and the id simply tells them apart.

6. Why threads appear: the listener loop

Here is the subtle part. The API will deliver nothing until you call app.run(), which starts an infinite loop that listens for incoming messages and dispatches each to the right callback. But an infinite loop blocks every line of code beneath it. The resolution is to run app.run() on a separate daemon thread, leaving your main thread free to keep firing requests. That is the entire reason threading enters an IB program — not cleverness, just the need to listen and talk at the same time.

Define the callback or lose the answer. If you forget to define currentTime(), the request still goes out and the server still replies — but your program has no method to catch the response, so nothing prints. The request and its callback are two halves of one whole; you always need both.
Two small but real gotchas. First, the flurry of messages TWS prints on startup are not errors — they're data-farm connection notices; ignore them. Second, do your live trading in Spyder or VS Code, not Jupyter: Jupyter is a research tool and isn't designed for the asynchronous, infinite-loop machinery that live streaming needs. (This study notebook only mocks that loop, so it runs safely.)

The one-paragraph version

To automate a validated strategy you connect a Python client to the broker's server (IB's TWS) through an API — here the ibapi package. Learn the operations manually first (TWS, paper trading). The API is asynchronous only, so you talk through two classes: EClient to send requests and EWrapper to receive responses via callbacks, paired one-to-(one-or-more). You combine them with multiple inheritance into a Strategy(EClient, EWrapper) class, create app, connect(host, port, clientId), and run the infinite app.run() listener on a daemon thread so the main thread can keep sending. Define every callback or the response vanishes; ignore the startup connection notices; and use Spyder/VS Code rather than Jupyter for the real thing.
