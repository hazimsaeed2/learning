① What a backtest really tells you

Not just "how much profit in the past" — it reveals the behaviour of a strategy: how it acts in different regimes, where it bleeds, how long it sits flat.

  • Goal: simulate real-world decisions, step by step, as if trading that very day.
  • Don't over-fit. Sweeping 50/200 → 90/10 → 80/40 crossovers to maximise past return = curve-fitting; the future won't repeat the past.
  • Options are the hardest asset to backtest — huge, messy, multi-dimensional data.

② Why you need the underlying too

A long call on its own can't be backtested — moneyness (ATM/ITM/OTM) and P&L both depend on where the underlying is.

④ Fix the data types

A number arriving as text breaks every calculation. The classic trap: IV stored as a string with a leading space.

c_iv = ' 0.1670' → str ✗
pd.to_numeric(c_iv.str.strip())
c_iv = 0.167 → float64 ✓
  • expire_date, quote_date → pd.to_datetime
  • c_* / p_* Greeks, IV, volume → pd.to_numeric
  • c_size = "7x7" bid×ask sizes — peculiar text, keep as-is

⑤ Classify by expiry (compute DTE yourself)

Most feeds omit days-to-expiry, so build it: dte = expire_date − quote_date, then tag each row.

⑦ The event-based engine — its DNA

Execution logic
when enter / exit
Time sensitivity
check every day
# walk day by day
if no position and day == first:
  legs = setup_long_call(day) # swap me!
  record entry premium
elif have position:
  mark-to-market today
  if day == day_before_expiry:
    exit, log round-trip

Event-based can carry stop-losses and go live; a vectorised backtest cannot. Swap one function — setup_long_call → setup_long_strangle → setup_long_butterfly — and the same engine backtests a new structure.

⑧ The reusable helpers

  • get_premium(day, strike, kind) — price of a contract, used as entry/exit.
  • atm_strike(day) — strike with the minimum distance to the underlying.
  • setup_*(day, direction) — returns the legs; direction='short' flips it.
  • mark-to-market — current P&L of the open position any day.

Three output frames fill as the loop runs: round-trip log, trades (entry/exit date, premium, P&L, exit reason), daily MTM.

  • Avg win +$5,821 vs avg loss −$2,135 — few big winners carry many small losers.
  • Strangle (same engine, swapped setup): net −$7,705 (50% win, PF 0.78) — the big-move bet lost in a grind-up year.
  • Max drawdown (trade equity): −9.83%. Beware: margin-normalised daily returns cry −99.9% / Sharpe −3.3 — an artifact; judge by P&L & profit factor.
  • Stop-loss −20% / take-profit +40%: 7 stops, 5 takes, net +$1,499 (avg hold cut to 2 days).

⑪ Extending the engine

  • Entry rule: add a column (e.g. iv ≥ 1.1×realized_vol, the variance-risk-premium) and gate entry on it.
  • Exits: add stop-loss / take-profit beside the expiry check → multiple exit reasons.
  • Sizing: never risk 100% on one trade; allocate least capital where daily returns are most volatile — predictability earns capital.

Remember

🧹 Clean first. Names, dtypes, spaces — garbage data ruins any backtester.

🔁 Event-based > vectorised. Day-by-day logic carries stop-losses and goes live.

📉 Judge the shape. Win rate, equity curve, max drawdown — not cumulative return or Sharpe alone.
