🏛️ Five must-haves of a serious algo engine
1 Event-driven — kills look-ahead bias by design; backtest code = live code
2 CEP engine — stateful: remembers past state, not just reacts to a tick
3 Quality data — survivorship-free + point-in-time + roll/corp-adjusted
4 Institutional APIs — complete, unambiguous, performant
5 Software engineering — the 95/5 split: ~95% of trading code is boilerplate (data, reconnects, reconciliation); only 5% is the strategy. Blueshift owns the 95% so you write the 5%.
🔁 Vectorized vs Event-driven — the central lesson
Vectorized (one pass)
  • read whole history → compute signals → P&L in one shot
  • + fast, great for first idea screening
  • − has past and future in one frame → look-ahead leaks
  • − stateful rules hard; production deploy is a rewrite
Event-driven (tick by tick)
  • bar arrives → react → maybe order → update state → wait
  • + look-ahead impossible by construction
  • + stateful rules natural; same code runs live
  • − slower (Blueshift uses an Apache-Arrow CEP engine)

🧱 The 3 pillars (any institutional engine)
  1. 1 Callbacks you register: initialize, before_trading_start, handle_data, scheduled fns. "If this event, invoke me."
  2. 2 Context — the algo's state object. Attach your own attributes (context.params=…) instead of globals; it's passed to every callback.
  3. 3 Domain APIs — order_target, data.history, symbol, set_stop_loss, schedule_function.

🎯 Orders — prefer target forms
AbsoluteTargeting (✓ recommended)
order(asset, n)order_target(asset, n)
order_value($)order_target_value($)
order_percent(%)order_target_percent(%)

Targeting trades only the difference. Hold 3, target 5 → buys 2. Hold 10, target 5 → sells 5. Avoids order "machine-gunning" when a condition stays true (e.g. RSI>70 firing every bar).

Return = order-id (str) · list of ids (retries) · None = failed.

⌨️ Blueshift strategy skeleton (reactive — declare functions registered to events)
from blueshift.api import symbol, order_target_percent, schedule_function, date_rules, time_rules

def initialize(context):              # MUST exist (else invalid strategy); runs once
    context.securities = [symbol('NIFTY-I'), symbol('BANKNIFTY-I')]
    context.params = {'lookback': 60, 'buy': 0.5}    # params on context, not globals
    set_commission(commission.NoCommission()); set_slippage(slippage.NoSlippage())
    schedule_function(run_strategy, date_rules.every_day(), time_rules.every_nth_minute(30))

def before_trading_start(context, data):   # ~30min pre-open, daily
    context.do_trade = True

def run_strategy(context, data):           # generate_signal → target → rebalance
    if not context.do_trade: return
    generate_signal(context, data); generate_target(context, data)
    for a in context.securities: order_target_percent(a, context.target[a])
🤖 LLM code-gen — focus on why, not how
  • Generate Blueshift code with ChatGPT/Copilot using the Developer-Kit LLM guides (GitHub) → fewer invented API calls, often one-shot.
  • You can never fully test a trading strategy (can't simulate all market states) → always walk through the code.
  • "Writing can be done by the LLM; your job is to understand the logic."
✅ Nine takeaways
  1. Blueshift = low-frequency, event-driven (idea, not wires).
  2. Five must-haves; the platform owns the 95% boilerplate.
  3. Event-driven beats vectorized — look-ahead-safe, stateful, deploys live.
  4. Three pillars: callbacks · context · domain APIs.
  5. Prefer order_target — trades only the difference.
  6. Data portal returns the simplest shape; strictly point-in-time.
  7. Quant lifecycle maps onto the code.
  8. Pipeline (factor/filter DAG) for large universes.
  9. Backtest ≠ truth → forward-test, and read the LLM's code.
