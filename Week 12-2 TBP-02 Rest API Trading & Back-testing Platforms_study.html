One Skill, Any Broker: Automating Trades with a REST API

Week 12-2 · Trading & Backtesting Platforms · TBP-02

In the previous session we automated trading through Interactive Brokers' TWS API — a custom Python library with its EClient and EWrapper classes. It works beautifully, but it only talks to IB. This session teaches a second, more universal route: the REST API. The trading operations we want — place an order, fetch positions, stream a price — stay exactly the same; only the technology for talking to the broker changes.

Why learn a second way at all? Two reasons. First, when you can accomplish a thing several ways, you make better engineering decisions about which is faster or simpler for a given job. Second — and this is the big one — REST has become the de-facto standard. The IT industry has run on REST for a decade, and since COVID almost every broker that offers algorithmic trading exposes a REST API: Indian, international, crypto, and FX brokers alike. Learn it once and you can connect to virtually any broker, even build a platform that fetches data from one broker and routes orders to another, or fails over to a backup when your primary disconnects.

What REST is not. It is not for high-frequency trading — it runs over the internet, where latency is measured in seconds, so it suits low- and medium-frequency trading (down to about one second). HFT firms use FIX and specialised FPGA hardware instead. REST is also not uniform (each broker exposes a different menu of operations — always read the docs) and not a replacement for your web or desktop terminal, which you keep for monitoring while REST runs the automation underneath.

1. From "API" to "REST API"

An API (Application Programming Interface) is simply a programmable bridge that lets a client talk to a server in a structured way. REST stands for Representational State Transfer, and here is the subtlety: REST is not a piece of software but a set of design principles — uniform interface, statelessness, cacheability, a client–server split. Any API built to honour those principles is called "a REST API." As the end user you needn't memorise the principles; what matters is that, unlike the IB library, a REST API arrives as plain HTTP URLs.

The name describes the flow exactly. Data lives on the server. The client requests the current state of some variable — your PnL, your positions, the last price — and the server reads that state and transfers a representation of it back. Hence Representational State Transfer. And it is synchronous: the client fires a request, waits, receives the response, acts, and only then sends the next one.

2. The five pieces of a request

Every REST request to a broker is assembled from five ingredients; omit one and the request fails:

#PieceWhat it carries
1Base URLthe broker server's address (on page one of the docs), e.g. https://localhost:5000/v1/api
2Endpointthe "department" for one operation, e.g. /pnl, /positions, /place_order
3HTTP methodyour intention: GET to fetch, POST to send — the docs tell you which
4Headersinformation about the client (auth token, content type) — a Python dict
5Payload / bodyinformation about the request itself (symbol, quantity, price) — sent as JSON

The full URL is just base URL + endpoint: .../v1/api/place_order. Think of the server as a bank — to talk about a loan you go to the loans desk, to deposit cash you go to the teller. Each endpoint is a desk that only handles its own kind of request.

3. The code pattern, on a mock broker

Because the rest of this lesson needs a live broker account, we stood up a tiny mock broker that behaves like IB's Client-Portal Web API — real status codes, real JSON — and fed it genuine NIFTY bars. The code reads exactly as it would against a real broker:

import requests, json
BASE = "https://localhost:5000/v1/api"

url = BASE + "/iserver/auth/status"     # pieces 1 + 2
headers, payload = {}, {}               # pieces 4 + 5 (none needed here)
r = requests.post(url, headers=headers, # piece 3: POST
                  json=payload, verify=False)
print(r.status_code)                    # response part 1
print(json.loads(r.content))            # response part 2
A real-world quirk. The very first time we check auth status it reports not authenticated — we haven't logged in. And IB drops idle sessions: in the live class the instructor logged in, spent a few minutes explaining the code, re-ran it, and found himself disconnected. That's a feature, not a bug. (Also, verify=False is IB-specific — its local gateway has no SSL certificate; other brokers use a hosted URL with proper SSL.)

4. Reading the response: status code first, then content

The response has two parts, and you always check them in order. First the status code, whose first digit tells you everything:

Starts withMeaningWhose fault
2xxsuccess — the server understood and processed your request—
4xxclient error — a bad URL, endpoint, method, header or payload (the famous 404 = page not found)you
5xxserver error — your request was fine but the server can't cope right nowthe broker

Only once the status code is in the 2xx range do you bother reading the second part, the content. These codes aren't broker inventions — they're an HTTP-consortium standard, identical across the entire web.

5. GET, POST, and placing an order

GET means "get me information" — fetching PnL or positions. POST means "here is new information to act on" — placing an order. A POST is where all five pieces earn their keep: the headers identify the client, and the payload carries the order details:

headers = {"Content-Type": "application/json"}            # about the client
payload = {"symbol": "NIFTY", "side": "BUY", "quantity": 75}  # about the request
r = requests.post(BASE + "/place_order",
                  headers=headers, json=payload, verify=False)
# -> status 200, {"order_id": 1001, "status": "Submitted"}

6. Why JSON is the common tongue

Your headers and payloads are Python dictionaries, but the server might be written in C++, which has no notion of a "dict." The neutral meeting ground is JSON — JavaScript Object Notation, which despite the name has nothing to do with JavaScript; it's just a text format every language can read and write. The json library bridges the two worlds: json.dumps(dict) turns a dict into JSON text to send, and json.loads(text) turns the server's JSON reply back into a dict. Send a dict, receive a dict, and JSON quietly handles the translation in between.

7. A modular library, and staying connected

The instructor's tidy design pattern: write one function per endpoint in a file like cp_web_api_functions.py — effectively your own little broker library — then import those functions into an entry_point.py where your strategy lives. That way your strategy code never has to know anything about base URLs or headers; it just calls fetch_historical() or place_order(). We mirrored exactly that.

The heartbeat. Since IB disconnects an idle session, you keep it alive either by calling the /reauthenticate endpoint or by running a continuous tickle — a small loop that pings a validate/tickle endpoint roughly every 55 seconds on a separate thread, just like the listener thread from TBP-01. One thing you may not automate is the initial login: regulations require a human to log in, so brokers deliberately don't expose that.

The one-paragraph version

A REST API is the broker-agnostic, de-facto-standard way to automate trading over HTTP — ideal for LFT/MFT, never for HFT (which uses FIX and FPGA). REST means Representational State Transfer: a synchronous client request for the state of some variable, returned as a representation. Build each request from five pieces — base URL, endpoint, HTTP method (GET to fetch, POST to send), headers (about the client) and payload (about the request, as JSON). Read the response in two parts: the status code (first digit — 2xx success, 4xx your error, 5xx server error) then the content. Use JSON (dumps/loads) as the common language with any server, keep the session alive with a tickle thread (you can't auto-login), and wrap one function per endpoint into your own library so your strategy code stays clean — exactly what we did against a mock broker fed with real NIFTY bars.
