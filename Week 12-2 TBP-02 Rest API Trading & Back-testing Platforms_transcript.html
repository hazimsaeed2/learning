3️⃣ REST API ARCHITECTURE — Base URL, Endpoints & HTTP Methods

WHAT: The Client-Server Architecture

WHY REST API? It's the de facto standard. Almost every broker post-COVID supports it: IB, Zerodha, Alpaca, crypto exchanges, forex brokers. Learn one technology → work with any broker.

THE "OFFICE DEPARTMENTS" ANALOGY:

• Base URL = the building address (broker's server)

• Endpoints = departments inside the building (positions dept, orders dept, PnL dept)

• To perform an operation, you go to the RIGHT department (endpoint)

• Wrong department → request rejected (404 error)

Base URLs by Broker:

BrokerBase URL
Interactive Brokershttps://localhost:5000/v1/api
Zerodha (India)https://api.kite.trade
Alpaca (US/Crypto)https://paper-api.alpaca.markets

HOW: Endpoints & HTTP Methods

OperationEndpointMethodIntention
Auth Status/iserver/auth/statusPOSTSend auth check
Place Order/iserver/account/{id}/ordersPOSTPost new order
Fetch Positions/portfolio/positionsGETGet data from server
Historical Data/iserver/marketdata/historyGETGet data from server
Fetch PnL/portfolio/pnlGETGet data from server
Re-authenticate/iserver/reauthenticatePOSTPost re-auth request

GET vs POST — THE SIMPLE RULE:

• GET = "Give me data" (fetch positions, prices, PnL)

• POST = "Here's new data" (place order, authenticate, modify alert)

• Don't memorize — the broker's documentation tells you which to use

⚠️ WARNING — REST API is NOT for HFT: REST operates over the internet with latency in seconds. HFT needs sub-millisecond latency using FIX (Financial Information eXchange) protocol or FPGA (Field-Programmable Gate Array) hardware. REST is for LFT/MFT only.

4️⃣ REQUEST DETAILS — Headers, Payload & Query Parameters

HEADERS = Client Info

WHAT: Metadata about your client — who you are, what format you accept, auth tokens.

# Example 1: Simple header
headers = {
    "Content-Type":       # What format we send
        "application/json"  # JSON format
}

# Example 2: With auth token
headers = {
    "Authorization":      # Auth credentials
        "Bearer abc123...",  # Token from broker
    "Content-Type":       # Data format
        "application/json",  # JSON
    "x-device-type":      # Device info
        "web"               # Web client
}

GOTCHA: Headers are Python dicts. Some endpoints need them, some don't — check documentation.

PAYLOAD/BODY = Request Info

WHAT: The actual trading data — instrument, quantity, price, order type. Sent in the request body.

# Place order payload
payload = {
    "conid": 265598,       # Contract ID (AAPL)
    "secType": "STK",      # Security type: stock
    "orderType": "LMT",    # Limit order
    "side": "SELL",        # Sell side
    "price": 250.50,       # Limit price
    "quantity": 30,        # 30 shares
    "tif": "DAY"           # Time in force
}
# MUST convert to JSON before sending:
data = json.dumps(payload) # Dict → JSON

⚠️ MUST json.dumps(): Payload must be converted to JSON before sending. Server doesn't understand Python dicts!

QUERY PARAMS = URL Additions

WHAT: Key-value pairs appended to the URL after ?. Alternative to payload for some endpoints.

# Fetch historical data — uses params
query_params = {
    "conid": 265598,       # Contract ID
    "period": "1y",        # 1 year of data
    "bar": "1d"            # Daily bars
}

response = requests.get(   # GET method
    url,                    # Base URL + endpoint
    headers={},             # Empty headers
    data="",                # Empty payload
    params=query_params,    # Appended to URL
    verify=False            # IB only
)
# URL becomes: ...?conid=265598&period=1y&bar=1d

💡 Payload vs Query Params: Payload is hidden in request body. Query params are visible in the URL to network devices. Same purpose, different visibility.

5️⃣ JSON — The Universal Data Format (Python Dict ↔ JSON Conversion)

WHAT & WHY: The Language Barrier Problem

WHY we need JSON: Your Python code sends a dict {"qty": 30}. The server might be coded in C++, Java, or Go — they don't understand Python dicts. JSON is the common format ALL languages understand.

THE TRANSLATION FLOW:

CLIENT (Python):

Python dict → json.dumps() → JSON string → SEND TO SERVER

SERVER (any language):

Processes request → Creates JSON response → SEND TO CLIENT

CLIENT (Python):

Receive JSON string → json.loads() → Python dict → USE IT

WORKED EXAMPLE:

Python dict: {"price": 250.50, "qty": 30}

→ json.dumps() → '{"price": 250.5, "qty": 30}' # STRING

→ send to server → server processes → sends back:

→ '{"orderId": "ABC123", "status": "filled"}' # STRING

→ json.loads() → {"orderId": "ABC123", "status": "filled"} # DICT

HOW: The Two Essential JSON Methods

import json                          # Built-in library

# ═══ SENDING: Python dict → JSON string ═══
payload = {"conid": 265598,          # Python dict
           "qty": 30,                # Native Python
           "side": "BUY"}            # Not sendable yet

json_string = json.dumps(payload)    # Convert to JSON
print(type(json_string))             # <class 'str'>
print(json_string)
# Output: '{"conid": 265598, "qty": 30, "side": "BUY"}'

# ═══ RECEIVING: JSON string → Python dict ═══
response_text = '{"authenticated": true}'  # From server
result = json.loads(response_text)         # Convert to dict
print(type(result))                  # <class 'dict'>
print(result["authenticated"])       # True (Python bool)

# ═══ PRACTICAL USAGE ═══
response = requests.post(url,        # Make request
    headers=headers,                  # Client info
    data=json.dumps(payload),         # dumps BEFORE sending
    verify=False)                     # IB only

status = response.status_code         # 200 = success
content = json.loads(response.text)   # loads AFTER receiving
pprint(content)                       # Pretty-print dict

🔑 MEMORY TRICK:

• dumps = "dump string" = dict → string (sending OUT)

• loads = "load string" = string → dict (loading IN)

6️⃣ MAKING REQUESTS — Complete Python Implementation

HOW: Auth Status Request (No Payload)

def fetch_auth_status():
    base_url = (                       # Broker address
        "https://localhost:5000/v1/api")
    endpoint = (                       # Auth department
        "/iserver/auth/status")
    url = base_url + endpoint          # Full URL

    headers = {}                       # No headers needed
    payload = {}                       # No payload needed

    response = requests.post(          # POST method (docs say)
        url,                           # Where to send
        headers=headers,               # Client info (empty)
        data=json.dumps(payload),      # Request info (empty)
        verify=False)                  # Skip SSL (IB only)

    status_code = response.status_code # 200 = success
    content = json.loads(              # Parse JSON response
        response.text)                 # Raw text → dict

    print(f"Status: {status_code}")    # Status: 200
    pprint(content)                    # Pretty-print dict
    # Output:
    # {'authenticated': True,
    #  'connected': True,
    #  'competing': False}

WORKED EXAMPLE — Reading the response:

status_code = 200 → starts with 2 → SUCCESS ✓

content["authenticated"] = True → we're logged in

content["connected"] = True → connection is active

HOW: Place Order Request (With Payload)

def place_order(account_id):
    base_url = (                       # Broker address
        "https://localhost:5000/v1/api")
    endpoint = (                       # Orders department
        f"/iserver/account/"
        f"{account_id}/orders")        # Account-specific
    url = base_url + endpoint          # Full URL

    headers = {                        # Client info
        "Content-Type":                # Format we send
            "application/json"}        # JSON format

    payload = {                        # Order details
        "conid": 265598,               # AAPL contract ID
        "secType": "STK",              # Stock
        "orderType": "LMT",           # Limit order
        "side": "SELL",                # Sell side
        "price": 250.50,               # Price per share
        "quantity": 30,                # 30 shares
        "tif": "DAY"}                  # Day order

    response = requests.post(          # POST (new data)
        url,                           # Full URL
        headers=headers,               # Client info
        data=json.dumps(payload),      # Order → JSON
        verify=False)                  # IB only

    status_code = response.status_code # Check result
    if status_code == 200:             # Success?
        pprint(json.loads(             # Print order conf
            response.text))
    else:                              # Error?
        print(f"Error {status_code}")  # Print error code
        pprint(json.loads(             # Print error detail
            response.text))            # Debug from here

7️⃣ WEB SOCKETS — Streaming Live Data (Prices, Orders, PnL)

WHAT & WHY: Synchronous vs Asynchronous

Synchronous (REST API): Client sends request → waits → gets response → sends next. Like calling a bank — ask one question, wait for answer, ask next.

Asynchronous (Web Sockets): Client subscribes once → broker pushes data whenever it changes. Like WhatsApp — messages arrive anytime without you asking.

WHEN TO USE EACH:

REST APIPlace/modify/cancel orders, fetch positions, auth, historical data
Web SocketsLive prices, order status updates, real-time PnL, market data streaming

WORKED EXAMPLE — Why async matters:

You have 100 limit orders open. Synchronous: poll status of each order every second = 100 requests/second → broker bans you. Asynchronous: subscribe to order updates once → broker pushes updates only when something changes → efficient.

💡 KEY INSIGHT: Your broker's mobile app does the same thing — when you login, it subscribes to live PnL via web sockets under the hood. When you build your own platform, YOU must subscribe explicitly.

HOW: Web Socket Implementation (Alpaca BTC Example)

import websocket                     # Web socket library
import json                          # JSON conversion
import ssl                           # SSL certificates

def on_open(ws):                     # Called when connected
    print("Connection opened")       # Log connection
    auth = {                         # Auth credentials
        "action": "auth",            # Auth action
        "key": "YOUR_API_KEY",       # Alpaca API key
        "secret": "YOUR_SECRET"}     # Alpaca secret
    ws.send(json.dumps(auth))        # Send as JSON

def on_message(ws, message):         # Called on each msg
    data = json.loads(message)       # JSON → Python dict
    if data[0]["T"] == "success":    # Auth successful?
        sub = {                      # Subscribe request
            "action": "subscribe",   # Subscribe action
            "quotes": ["BTC/USD"]}   # Bitcoin quotes
        ws.send(json.dumps(sub))     # Send subscription
    else:                            # Price data arrived
        pprint(data)                 # Print live quotes

def on_error(ws, error):             # Called on error
    print(f"Error: {error}")         # Log the error

def on_close(ws, code, msg):         # Called on disconnect
    print("Connection closed")       # Log disconnect

# Create & run the web socket
wss_url = (                          # Alpaca WSS endpoint
    "wss://stream.data.alpaca.markets"
    "/v1beta3/crypto/us")            # Crypto stream URL
ws = websocket.WebSocketApp(         # Create app
    wss_url,                         # Stream URL
    on_open=on_open,                 # Connect callback
    on_message=on_message,           # Message callback
    on_error=on_error,               # Error callback
    on_close=on_close)               # Close callback
ws.run_forever()                     # Listen forever
# Runs infinite loop — use threading for background

• base_functions.py — All broker-specific REST API code (functions for each endpoint)

• entry_point.py — Strategy/business logic that imports from base_functions

🎯 KEY TAKEAWAYS

1. Every REST API request needs 5 things: Base URL + Endpoint + HTTP Method + Headers + Payload. Missing any one → request fails. Always check the broker's documentation first.

2. JSON is the universal translator: json.dumps() converts Python dicts to JSON before sending, json.loads() converts JSON back to Python dicts after receiving. Never send raw Python dicts.

3. Use REST API for trading operations (orders, positions, PnL) and Web Sockets for live streaming data (prices, order updates). REST = synchronous, Web Sockets = asynchronous.

4. Build modular code: broker-specific functions in one file, strategy logic in another. To switch brokers, rewrite only the broker file. REST API is the de facto standard — same pattern works everywhere.

THE MASTER FORMULA: response = requests.post(base_url + endpoint, headers=headers, data=json.dumps(payload)) → check response.status_code → parse json.loads(response.text) → use the data

THE ONE THING TO NEVER FORGET: REST API uses synchronous communication (request → wait → response). For live data streaming, use Web Sockets (subscribe once → broker pushes updates). And ALWAYS json.dumps() before sending, json.loads() after receiving.
