Machine Learning-I: Foundations, Trees & the Logistic Classifier

A plain-English walk through Jay's first ML session — what AI, ML and DL actually mean, the three jobs machine learning does, how decision trees split data by purity and grow into random forests, how a logistic regression turns a straight line into a probability, and how to build a complete trading classifier on real Tesla data without quietly cheating.

This is the opening of the machine-learning module, and the instructor is deliberate about it: the field has grown from a topic into an entire discipline, so rather than derive every equation, these first sessions build intuition for what the tools are and how they're used in trading. The promise is that once the foundation is solid, later lectures slot real strategies on top of it.

1. AI, ML and DL — three nested ideas

The instructor's favourite illustration is a doctor in a remote town. She sees familiar cases easily, but when an outsider arrives with unfamiliar symptoms she sends them to the city for tests, then files the report with a label — "this person had X". Collect thousands of such labelled reports from many such doctors, feed them to a machine, and you get a system that can read a new report and suggest a diagnosis. That system is artificial intelligence: it makes a decision a human would make, because it was built from human (natural) intelligence captured as labelled history.

Inside AI sit two families of algorithms. Machine learning is the set of algorithms that learn the association between inputs and outputs from data. Deep learning is a subset of ML inspired by how the brain decides (neural networks), covered later in the module. So the nesting is simply: AI ⊃ ML ⊃ DL. And a reassurance the instructor offers: AI is here to make us more productive and help us decide better — not to replace the natural intelligence that built it.

2. The three jobs machine learning does

Everything in this module reduces to one of three tasks:

TaskWhat it predictsHas labels?Family
Regressiona number (price, volatility, return)yesSupervised
Classificationa class (up/down, cancer/not)yesSupervised
Clusteringa group id (similar items together)noUnsupervised

Two of these — regression and classification — are about forecasting, and because we supply the right answers (labels) to learn from, they are supervised. Clustering is about analysis: we have no labels, we just want to discover structure, so it is unsupervised (as is dimensionality reduction, which compresses many columns into a few without losing much information). A third style, reinforcement learning, gets its own dedicated class later.

3. Classification, told as training a child

Imagine teaching a toddler to tell apples from oranges. The child instinctively explores each fruit's features — weight, colour, taste, shape — while you, the supervisor, supply the label: "this is an apple". After enough examples the child forms an internal association between the features and the label, and can name a new fruit unaided.

A dataset works exactly the same way. Each row is an observation, each column is a feature, and one special column is the target (the label). In trading the features are technical or statistical properties derived from OHLC data, and the target is something like up(1)/down(0), or buy/sell/hold. Crucially, we design the target — and how we define it (any positive return? only a 5% move?) decides what the model learns. That act of supervising the target is precisely why these are called supervised methods.

4. Logistic regression: bending a line into a probability

Start with ordinary linear regression. With a numeric target — say predicting price from RSI — we fit the line of best fit z = α·RSI + β, the line that minimises the squared distances to the points. It captures the data's shape nicely.

Now change the target to a binary 0/1 (did the stock go up?). The cloud of points splits into two horizontal bands, and no straight line can represent it — the errors are huge no matter how you tilt the line. The fix is elegant: take that same linear score z and pass it through the sigmoid (a.k.a. logit) function:

σ(z) = 1 / (1 + e−z)

This squashes any number into the range (0,1) and bends the line into a smooth S-curve. The output is now read as a probability. A handful of values make the shape obvious: σ(0)=0.50 (dead centre), σ(2)=0.88, σ(−2)=0.12, saturating toward 1 and 0 at the extremes. We then apply a threshold — 0.5 by default, though we are free to raise it — so that a probability above the threshold means class 1, below it class 0.

The naming trap. Despite "regression" in its name, logistic regression is a classification algorithm. For more than two classes there are extensions (multinomial / softmax), but the workhorse version classifies into two.

5. Decision trees, purity, and the forest

The other classifier this lecture builds is the decision tree — a flowchart of yes/no questions. At each node the tree picks one feature and one threshold that best separates the classes, and keeps splitting until the branches are "pure enough". The instructor illustrates it with a 30-person dataset recording each person's calorie intake, exercise duration and age, labelled Fit (yes/no). A first split on Age < 27 already carves out a very clean branch.

How does the tree know a split is good? It measures impurity before and after. The lecture's workhorse is the Gini impurity, Gini = 1 − Σ pj2, where pj is the fraction of class j at a node. A node with only one class scores 0 (perfectly pure); a 50/50 node scores 0.5 (maximally impure).

The worked example. The full 30 people are 20 Fit / 10 Unfit, so the parent Gini is 1 − (20/30)² − (10/30)² = 0.4444 (the slide rounds this to 0.4422). Split on Age < 27: the younger branch holds 7 people (6 Fit, 1 Unfit) with Gini 1 − (6/7)² − (1/7)² = 0.2449; the older branch holds 23 people (14 Fit, 9 Unfit) with Gini 0.4764. Weighting each branch by its size gives 0.2449·(7/30) + 0.4764·(23/30) = 0.4224. Impurity fell from 0.4444 to 0.4224, so the split is worthwhile — and the tree greedily chooses whichever split reduces impurity most.

Gini is not the only ruler. Entropy (and the resulting information gain) powers the ID3 and C4.5 algorithms, and plain classification error 1 − max(pj) is a third option; all three measure the same idea of impurity. A deep tree can memorise almost any training set, which is why Occam's razor guides us: between two models of similar accuracy, prefer the simpler (shallower) one. We enforce that with pruning — either pre-pruning (stop growing once the tree gets large) or post-pruning (grow fully, then trim weak nodes bottom-up).

A single tree is unstable and overfits, so we combine many into a random forest. Each tree is grown on a different bootstrap sample of the data and, at each node, is allowed to consider only a random subset of the features. The forest then votes — the majority class wins. Because the individual trees are de-correlated, their errors partly cancel, and the forest generalises noticeably better than any one tree.

6. Building a real trading classifier — without cheating

The lecture then builds an end-to-end model whose goal is stated carefully: predict whether the next day is an up-day or NOT (note: up vs not-up, not up vs down). The practice notebook reproduces every step on the real Tesla (TSLA) daily series shipped with the class — 1510 rows from the end of 2013 onward.

Step 1 — Build the target with shift(-1)

Compute log returns, then label 1 when tomorrow's return is positive: np.where(log_ret.shift(-1) > 0, 1, 0). On the real Tesla data this gives 761 up-days versus 749 not-up — a base rate of about 50.4%, almost a coin toss. A natural worry: isn't peeking at tomorrow look-ahead bias? No — we are merely constructing the dataset the machine will later learn from. The look-ahead rule applies to training and testing, which haven't happened yet.

Step 2 — Engineer features from OHLC

Raw Open/High/Low/Close/Volume is the same data everyone has, so the edge comes from feature extraction: rolling volatility (standard deviation over windows of 5, 10 and 15 days), trend (rolling means over 10, 15, 20 and 25 days), momentum (summed percentage returns over 3, 6 and 9 days), a 4-day volume moving average, and the intraday move Close − Open — twelve features in all. Rolling windows leave NaNs at the start, which we drop, leaving 1485 usable rows. The instructor's emphasis: features make or break a model — start from a hypothesis about what should predict the target rather than throwing everything into a "kitchen sink".

Step 3 — Split, but never shuffle

We split into 75% training and 25% testing with train_test_split(..., test_size=0.25, shuffle=False). The shuffle=False is essential: this is a time series, so the earliest 75% must train and the latest 25% must test. Shuffling would let the model train on the future to predict the past — the very definition of look-ahead bias.

Step 4 — Scale, learning the rule only from training data

The features live on wildly different scales — a volume average in the tens of millions next to a momentum figure near zero. Left raw, the model would lazily assume the big numbers matter most. The z-score (x − μ)/σ via StandardScaler rescales every feature to mean 0 and standard deviation 1, without changing its shape or distribution.

The cardinal mistake. Call fit_transform on the training set (which learns μ and σ from training data), then transform — never fit — the test set, reusing the training statistics. Fitting the scaler on the test set leaks future information and silently inflates your accuracy. This is the single most common way people fool themselves. (Trees, being scale-invariant, skip this step; logistic regression needs it.)

Step 5 — Train three models, and stay honest about the score

We fit all three classifiers on the same split, each with a fixed random_state for reproducibility. The decision tree (max_depth=3) scores about 56% on training but only 49.7% on the unseen test set — a textbook case of a tree overfitting. The random forest (20 shallow trees, voting) lifts test accuracy to 51.9%, and the logistic regression reaches 53.0% — the best of the three, and still only a whisker above the 50.4% base rate. The tree's feature-importance scores reveal it leans hardest on the medium-term moving average ma_15. The predict_proba method shows the two class probabilities per row (they always sum to 1); whichever exceeds the threshold wins. Accuracy is just one metric — the next lecture is devoted to evaluating models properly.

7. The other side: clustering

To see the unsupervised half, picture 50 companies plotted by return and volatility, with no labels at all. We hand only the features to K-Means, tell it how many clusters to find (it cannot guess that for us), and it returns group assignments — in the notebook, roughly 17/16/17 across three clusters. The machine groups; we supply the meaning afterward (perhaps low-vol versus high-vol names). In trading, this drives portfolio construction and the grouping of open positions by risk exposure for real-time risk monitoring.

8. The takeaways

  • AI ⊃ ML ⊃ DL; ML learns the association between features and labels from data.
  • Three tasks: regression (number), classification (class), clustering (group, no labels).
  • Decision trees split on the feature that most reduces Gini impurity (1 − Σ pj2); the worked example fell 0.4444 → 0.4224. They overfit, so prune and prefer simpler trees (Occam's razor).
  • Random forests grow many de-correlated trees (bootstrap + random features) and vote — generalising better (test 51.9% vs the tree's 49.7%).
  • Logistic regression = linear score → sigmoid → probability → 0.5 threshold → class. It classifies, and it needs scaling.
  • Build the target with shift(-1) before training (not look-ahead); engineer features from OHLC.
  • Split time series with shuffle=False; scale by fitting on train only and transforming test.
  • ~53% test accuracy is normal for next-day direction — features are the edge, and evaluation comes next.

The next session opens with how to evaluate these models properly, then broadens into more algorithms.
