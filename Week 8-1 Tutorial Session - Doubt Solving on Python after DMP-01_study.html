From Raw Data to a Working Strategy

Week 8-1 · A doubt-solving tutorial after DMP-01 — getting data in, and building your first moving-average crossover

This tutorial, led by Manusha Rao, is the practical companion to the DMP-01 back-testing lecture. Where that lecture explained the concepts, this session walks through the plumbing: how to actually pull market data into Python without tripping over the common errors, how to save and reload it, how to fix time zones, and finally how to assemble a complete moving-average crossover strategy and judge it against simply buying and holding. We follow along on the same NIFTY-50 daily file (962 bars, 2017 to 2020) used in class.

The libraries you'll keep reaching for

Every quant notebook opens with the same handful of imports. pandas holds and manipulates your data as DataFrames; numpy does fast array maths; datetime builds and handles dates; yfinance downloads price data from Yahoo Finance; warnings lets you silence the harmless messages Python sometimes prints; and matplotlib draws charts. The tutorial introduces one new name — cufflinks, a small bridge that lets a pandas DataFrame produce interactive Plotly charts directly with .iplot(), where matplotlib gives you a static picture.

A date is a date — not a tuple, not a string

The first live mistake everyone makes is this. You write start = (2017, 1, 1), which looks like a date but is actually a tuple (round brackets always make a tuple). Hand that to yf.download and it complains: 'tuple' object has no attribute 'timestamp'. A string like "2017-01-01" is no better — Python sees text, not a calendar date. The fix is to build a genuine date object:

import datetime as dt
start_date = dt.date(2017, 1, 1)
end_date   = dt.date(2020, 11, 27)

Now type(start_date) reports date, and yfinance is happy.

Two flags that save hours

The download call hides two important switches. The first is multi_level_index=False. Recent yfinance versions return columns with two levels — ('Close', '^NSEI') — which makes ordinary column selection a headache. Setting this flag flattens them back to plain Close, High, and so on. The second is auto_adjust. With it left at its default True, you get prices already adjusted for corporate actions. Set it to False and yfinance adds a separate Adj Close column while leaving the OHLC prices unadjusted.

import yfinance as yf
df = yf.download("^NSEI", start=start_date, end=end_date,
                 multi_level_index=False, auto_adjust=True)
Intraday has a hard limit. You can ask for 5-minute bars with interval="5m", but Yahoo only serves intraday data for the last 60 days. Ask for 61 and it errors. For longer intraday history you go to a (usually paid) broker feed.

What "adjusted close" actually means

It is worth pausing on this because it confuses people. The adjusted close folds in every corporate action — dividends, stock splits, share issuances, mergers — and applies the correction backwards across all prior history, so that a return series doesn't show a fake 50% drop on the morning a stock splits two-for-one. For a single stock this matters a lot. For an index like NIFTY-50 there are no splits or dividends in the same sense, so its adjusted close and ordinary close are essentially identical. The general rule: for long-horizon analysis, use the adjusted close.

Saving and reloading: the CSV round-trip

The DMP-01 lecture's contingency tip was "save your data to a CSV." Here's the mechanic. Writing is one call, df.to_csv("data.csv"), and the filename can be anything. Reading back is where a subtlety lives. A plain pd.read_csv("data.csv") gives you a DataFrame whose index is a meaningless RangeIndex (0, 1, 2, …) and whose Date is just another column — Python has no idea that column is special. You have to tell it:

df = pd.read_csv("data.csv", index_col="Date", parse_dates=True)

index_col promotes the Date column to be the index (you can name it, or give its position like index_col=0), and parse_dates=True turns those date strings into a real DatetimeIndex. With the plain read our file showed 6 columns and a RangeIndex; with these two arguments it became 5 columns with a proper time index — exactly what we want for a time series.

Windows paths bite. A backslash begins an escape sequence in Python, so a copied path like "C:\Users\me\data.csv" silently breaks. Either prefix the string with r to make it raw — r"C:\Users\me\data.csv" — or double every backslash. (When the file sits in the same folder as your notebook, just use its bare name.)

Time zones

Intraday timestamps from yfinance arrive in UTC — you'll see a +00:00 on the index. To read them in Indian time you convert:

intraday.index = intraday.index.tz_convert("Asia/Kolkata")
# 09:30 UTC becomes 15:00 IST

To discover the right zone name, the lecture uses the pytz library and its all_timezones list; Python's built-in zoneinfo.available_timezones() does the same job and lists nearly 600 names. A handy shortcut for any library: type the module name, a dot, and press Tab to see what's available — and append a ? to any function (yf.download?) to read its help.

Vectorization, and why it foreshadows event-driven testing

Here is a small experiment that carries a big idea. Take a plain Python list [1, 2, 11, 5, 8] and multiply it by 3. You might expect each element to triple; instead the list is repeated three times. Lists don't do element-wise maths. Convert it to a numpy array first, and array * 3 gives [3, 6, 33, 15, 24] — every element multiplied at once. That "all at once" is vectorization, and it's why numpy is fast.

If you wanted the same result from the original list, you'd write a loop: [v*3 for v in lst], processing one element at a time. That loop is exactly the shape of event-driven back-testing — stepping through the data candle by candle — whereas the array operation is the shape of vectorized back-testing, computing the whole series in one stroke. Same answer; very different speed and very different realism.

The payoff: a moving-average crossover strategy

Now everything comes together. The strategy is the classic crossover of two moving averages: a short one over 21 days and a long one over 63 days. The idea is momentum — when the short average climbs above the long average (a golden cross), the recent trend is up, so go long; when it drops below (a death cross), go the other way. We build it in seven vectorized steps.

d["SMA"]    = d["Close"].rolling(21).mean()      # 1. short MA
d["LMA"]    = d["Close"].rolling(63).mean()      # 2. long MA
d["regime"] = np.where(d.SMA > d.LMA, 1, -1)     # 3. +1 long, -1 short
d["signal"] = d["regime"].diff()                 # 4. +2 golden, -2 death
d["pos"]    = d["regime"].shift(1)               # 5. trade next bar
d["strat"]  = d["pos"] * d["Close"].pct_change() # 6. strategy returns
equity      = (1 + d["strat"]).cumprod()         # 7. equity curve

Step by step: rolling(21).mean() is the moving average (the first 20 rows come out empty — you need 21 values to average 21 days, and 63 for the long one, so we drop the warm-up). np.where turns the comparison into a regime: +1 whenever the short MA is above the long, −1 otherwise. Taking .diff() of that regime reveals the moments of change — a jump of +2 marks a golden cross, −2 a death cross. Then the by-now-familiar .shift(1) lags the position one bar so we trade on information we actually had, multiplying position by daily returns and compounding with cumprod.

The result

After the 63-day warm-up we have 900 rows. Over the window there were four golden crosses and four death crosses; the strategy was long about 692 days and short about 207. And the scoreboard:

ResultTotal return
Buy & Hold40.39%
SMA21 / LMA63 crossover68.04%

This time the strategy wins — and decisively. The reason is instructive: because the crossover can go short, the death cross flipped it to a short position heading into the February–March 2020 COVID crash, so it profited (or at least sidestepped the damage) while a buy-and-hold investor rode the index straight down. That is a genuinely good outcome on this data — but the honest caveat from the lecture still stands: it's a single vectorized back-test over one period on one instrument, with no slippage or fills modelled. Before trusting it with money you'd re-run it event-driven, on out-of-sample data, and across several instruments. The win is encouraging, not conclusive.

The one-paragraph version. Build dates with datetime, and remember yfinance's two flags (multi_level_index=False, auto_adjust) and its 60-day intraday limit. Save with to_csv and reload with read_csv(index_col=..., parse_dates=True), using raw strings for Windows paths. Convert intraday UTC timestamps to your zone with tz_convert. Lean on numpy's vectorization (array maths runs all at once; loops are event-driven). Then build a crossover strategy in seven clean steps — rolling means, a regime via np.where, crossings via .diff(), a one-bar .shift(), returns, and a compounded equity curve — and always hold it up against buy-and-hold to see whether it earned its complexity.
