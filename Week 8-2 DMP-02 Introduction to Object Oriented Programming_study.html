Object-Oriented Programming, From Scratch

Week 8-2 · Data Analysis & Modeling in Python · DMP-02 · with Ashutosh Dave

Up to now you have written code the way most people first learn it: one line after another, top to bottom. You imported a library, picked a ticker, ran a cell, then the next, then the next. That style is called sequential or functional programming, and it is perfectly fine when you only have a little to do. But open almost any serious project on GitHub, or read any professional's code, and you will see something different: code organised into classes and objects. That style is called object-oriented programming, or OOP, and learning it is the goal of this lesson.

We will do it in two halves. First we build the concepts on a tiny, friendly example — a Student. Then we put those concepts to work by turning a moving-average back-test into a clean, reusable class. The aim throughout is intuition first, syntax second.

1. Dunders: the magic you've been using all along

Before classes, one piece of vocabulary. A dunder — short for "double underscore" — is a built-in method whose name is wrapped in two underscores, like __add__ or __len__. They look mysterious, but they behave like any other method. Here is the trick: the ordinary things you do every day secretly call them.

2 + 2            # 4
(2).__add__(2)   # 4  — exactly the same thing

len([1, 2, 3])       # 3
[1, 2, 3].__len__()  # 3  — again, the same

When you write +, Python quietly calls __add__ behind the scenes. When you call len(), it calls __len__. Libraries are full of them — numpy.__version__ stores the version string, for instance. Dunders may look scary the first time, but they are just functions. The reason we meet them now is that one dunder in particular — __init__ — is the beating heart of object-oriented programming.

2. The big idea: blueprints and the things built from them

Imagine you are a designer and you have to define the concept of a car. You would say: to count as a car, a thing needs four tyres, a covered body, and an engine — those are its properties. It also needs to be able to do things: drive, sound a horn — those are its functionalities. That whole concept — the idea of a car — is what we call a class. It is a blueprint.

Once the blueprint exists, you can produce specific cars from it: a blue Mercedes, a black BMW, each with its own engine number. Those specific cars are objects, also called instances. Every car on the road is an instance of the class "car".

In Python, everything is already an object. When you write a = 4.3 and check type(a), Python answers <class 'float'> — your number is an object of the class float, and it comes with functionalities like (4.0).is_integer(). Lists, tuples, functions — all of them are objects built from classes that the Python authors wrote for you. The exciting part is that you can write your own classes too.

3. Building a class: shared properties and class methods

Let's define our Student. We start with the class keyword, a name, a colon, and a body. The first thing inside is goal, which looks like a plain variable set to "to get educated". A variable defined directly in the class body is a class attribute — it is shared by every student, because in general every student has that goal.

class Student:
    goal = "to get educated"          # class attribute (shared by all)

    @classmethod
    def update_goal(cls, new_goal):
        cls.goal = new_goal           # changes the goal for the WHOLE class

The function has two unusual features. On top sits @classmethod, a decorator. A decorator wraps a function to change how it behaves without rewriting its body; here it simply links the method to the Student class. Second, its first parameter is cls — by convention, the name for "the class itself". So when we call Student.update_goal("..."), it reaches into the class and rewrites goal for every student at once. That shared nature is the whole point of a class attribute.

4. Per-object data: __init__ and self

A shared goal is fine, but students also have things that are theirs alone — a name, an age. These belong on the instance, not the class. That is the job of the special dunder __init__, the constructor. It runs automatically the moment you create an object, and its first parameter, self, stands for the object being created.

class Student:
    goal = "to get educated"

    def __init__(self, name, age):    # runs automatically on creation
        self.name = name              # instance attribute
        self.age  = age

    def update_age(self, new_age):    # instance method
        self.age = new_age
        return f"new age for {self.name} is {self.age}"

john = Student("John Paul", 19)
john.name    # 'John Paul'
john.age     # 19

When you write Student("John Paul", 19), Python looks for __init__, sees it needs a name and an age, and assigns them: self.name becomes john.name, because here self is John. Those values, attached to the object, are instance attributes. A method whose first parameter is self is an instance method: it has full knowledge of that object's data and works on it.

Once you add an __init__ with required arguments, you must supply them. You can no longer write Student() — Python will complain about missing arguments. You have to give it the data it asked for: Student("Mary", 20).

The third kind: static methods

There is one more flavour. A static method, marked with @staticmethod, takes neither self nor cls. It knows nothing about the object or the class — it is just an ordinary helper function that you choose to keep under the class's umbrella for tidiness (printing the academic year, say, or a strategy's run-date). Here is the cheat for telling the three apart:

KindDecoratorFirst argumentKnows about
class method@classmethodclsthe class
instance methodnoneselfthe object
static method@staticmethodnonenothing

5. Inheritance: reusing a blueprint

The real power of OOP is that once you have a blueprint, you can make new blueprints from it. Suppose we want a SportyStudent. Rather than rewrite everything, we let it inherit from Student and change only what differs.

class SportyStudent(Student):                 # inherits everything
    goal = "to get educated and win some trophies"   # override just this

dave = SportyStudent("Dave", 17)
dave.update_age(18)     # 'new age for Dave is 18'  — inherited for free!
dave.goal               # '...win some trophies'

Student is the parent (or super-class); SportyStudent is the child (or sub-class). The child gets every attribute and method of the parent without a single line of new code, then diverges only where we want it to. A neat real-world example a student offered in class: pandas is, conceptually, built on top of numpy in just this way. Two helpers let you inspect the relationships:

isinstance(dave, SportyStudent)      # True
isinstance(dave, Student)            # True  — because the parent is in its ancestry
issubclass(SportyStudent, Student)   # True

super() and the family tree

Often a child wants its own constructor but also wants to run the parent's. That is what super() is for:

class Child(ParentOne, ParentTwo):
    def __init__(self, grade, age):
        self.grade = grade
        super().__init__(age)        # also run ParentOne's __init__

Child.__mro__   # Child -> ParentOne -> ParentTwo -> object

A class can even inherit from several parents at once. When it does, Python uses the MRO — the method resolution order — to decide which parent to search first. The MRO is simply the family tree, and it always ends in object, the common ancestor of every class in Python. You can see it for anything: a list's MRO is list → object; a DataFrame's is a long chain (DataFrame → NDFrame → … → object) precisely because a DataFrame is a rich, much-inherited structure. (One quick aside you might also use: dave.__dict__ shows an object's instance attributes as a dictionary.)

6. Putting it to work: a back-test as a class

Now the payoff. You already know the seven logical steps of a back-test: start with an idea, get the data, build indicators, generate signals, derive positions, compute returns, and finally analyse. In the sequential style you run those as cells, one after another. That is fine for one strategy — but clumsy when you want to test many. OOP lets us bundle all seven steps under a single umbrella.

class BacktestingCrossover:
    def __init__(self, csv, ma_short, ma_long):
        self.ma_short, self.ma_long = ma_short, ma_long
        self.fetch_data()      # each step is an instance method
        self.indicators()
        self.signals()
        self.positions()
        self.returns()

    def signals(self):
        self.df["regime"] = np.where(self.df["SMA_s"] > self.df["SMA_l"], 1, -1)
    # ... fetch_data / indicators / positions / returns similar ...

The __init__ stores the parameters as instance attributes — exactly like self.name and self.age earlier — and then calls each step in order. Every step is an instance method, so it already has access to self.df and the parameters. The beauty is what happens next: one object is one back-test. Creating a second back-test with different windows is a single line.

The result on real NIFTY data (962 daily bars, 2017→2020). Two objects from one blueprint, differing only in their short window:

Be honest about the numbers. Both crossover variants trail Buy & Hold badly over this period — the long/short regime goes short during a strong up-trend and gives back the gains. The lesson of Part 2 is structure, not alpha: notice how cleanly OOP lets us spin up and compare two strategies. Between the two, the faster SMA5/20 beat SMA10/20, but neither is something you'd trade as-is.
Why bother, then? Because when you later have ten ideas to test, the class pays off enormously. Each test becomes one object, the seven steps run themselves, and your notebook stays a short list of clean comparisons instead of a sprawling wall of copy-pasted cells.

The one-paragraph version

Object-oriented programming organises code around classes (blueprints) and objects (specific things built from them). The operators you already use are powered by dunders like __add__ and __init__. Inside a class, a class attribute is shared by all instances, while instance attributes — set on self in __init__ — belong to each object. Methods come in three kinds: @classmethod (gets cls), plain instance methods (get self), and @staticmethod (gets neither). Inheritance lets a child class reuse a parent's code and override only what differs; super() runs the parent's constructor, and the MRO (ending in object) resolves multiple parents. Wrapping a back-test's seven steps in a BacktestingCrossover class turns each run into a single clean object — which is exactly why production trading code is written this way.
