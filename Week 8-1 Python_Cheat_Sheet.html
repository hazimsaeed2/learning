📌 THE SHIFT(2) RULE — Why We Do It

DayEventCan Trade?
Day 4 Jan 4 Signal generated at close ($134.81) ❌ No
Day 5 Jan 5 Buy order executes at close ($135.00) ✅ Trade opens
Day 6 Jan 6 Return = (136.03−135.00)/135.00 Daily return

Signal (Day 4) → position reflected on Day 6 return = 2 days offset → use .shift(2)

Skipping shift(2) = Look-Ahead Bias — pretending you traded at prices you couldn't know yet. Backtest looks great but fails in real life.

5️⃣ LISTS vs NUMPY ARRAYS — The Foundation of Vectorization

The Problem with Python Lists

my_list = [1, 2, 5, 11, 22, 3, 8]

my_list * 3
# ❌ WRONG result — Python repeats the whole list 3 times:
# [1, 2, 5, 11, 22, 3, 8,  1, 2, 5, 11, 22, 3, 8,  1, 2, 5, 11, 22, 3, 8]

# To multiply each element you need a for loop — SLOW:
result = []
for val in my_list:
    result.append(val * 3)
# → [3, 6, 15, 33, 66, 9, 24]   ✅ correct, but row-by-row (slow for 900+ rows)

The Solution: NumPy Arrays (Vectorized)

import numpy as np

my_array = np.array([1, 2, 5, 11, 22, 3, 8])  # Convert list → array

my_array * 3
# ✅ CORRECT — multiplies every element simultaneously in one operation:
# [ 3,  6, 15, 33, 66,  9, 24]

This is VECTORIZATION — all elements processed at once, concurrently.

In backtesting: When you write np.where(condition_on_dataframe, 1, 0), NumPy checks all 899 rows simultaneously — not one at a time. This is vectorized backtesting and it's why it's fast.

💡 Event-driven = the for-loop version: Check each row one by one. Same result, but much slower. See Section 8 for when to use each.

6️⃣ THE FULL MOVING AVERAGE CROSSOVER STRATEGY — 9 Steps From Start to Finish

STEP 1 — Download Data

import datetime as dt
import yfinance as yf

start = dt.date(2017, 1, 1)   # datetime.date object
end   = dt.date(2020, 1, 1)   # (not a string — yf accepts both)

data = yf.download(
    "^NSEI",
    start=start,
    end=end,
    multi_level_index=False
)
# Result: 962 rows × 5 columns (OHLCV)
# index: DatetimeIndex starting 2017-01-02

STEP 2 — Calculate Moving Averages

short_w = 21   # Short window (days)
long_w  = 63   # Long window (days)

# .rolling(n) = sliding n-day window
# .mean()     = average of each window
data['SMA'] = (data['Close']
               .rolling(short_w)
               .mean())

data['LMA'] = (data['Close']
               .rolling(long_w)
               .mean())
# First 62 rows → NaN (not enough history)

Rolling = sliding window:

Days 1–21 → avg = SMA[21]

Days 2–22 → avg = SMA[22]

Days 3–23 → avg = SMA[23] ...

STEP 3 — Get Yesterday's Values

# .shift(1) = move column DOWN by 1 row
# Effect: today's row now ALSO has
#         yesterday's SMA/LMA value
data['SMA_prev'] = data['SMA'].shift(1)
data['LMA_prev'] = data['LMA'].shift(1)

# Crossover needs both today AND yesterday
# in the same row → that's what shift gives
DateSMA todaySMA_prev (yesterday)
Jan 18100NaN
Jan 281508100 ←
Jan 382008150 ←

STEP 4 — Drop NaN Rows

data.dropna(inplace=True)
# Removes first 63 rows (LMA needs 63
# days of price history before it starts)
# 962 rows → ~899 rows

⚠️ inplace=True is critical:

Without it, dropna() returns a new DataFrame but does NOT change data. You don't get an error — the NaN rows silently stay and corrupt your calculations later.

STEP 5 — Generate Signals

# np.where(condition, value_if_true, if_false)
# Processes ALL rows at ONCE (vectorized)

# Long (buy) = 1 when crossover UP
data['signal'] = np.where(
    (data['SMA'] > data['LMA']) &
    (data['SMA_prev'] < data['LMA_prev']),
    1,              # ← assign 1 (buy)
    0               # ← assign 0 (no signal)
)
# Short (sell) = -1 when crossover DOWN
data['signal'] = np.where(
    (data['SMA'] < data['LMA']) &
    (data['SMA_prev'] > data['LMA_prev']),
    -1,             # ← assign -1 (sell)
    data['signal']  # ← keep existing value
)
# Result: 4 ones, 4 minus-ones, rest zeros

STEP 6 — Build Positions (Hold)

# Forward fill: replace 0s with the last
# non-zero value (hold until next signal)
data['position'] = (data['signal']
    .replace(0, method='ffill'))

What forward fill does:

signal:0 0 0 1 0 0 -1 0
position:0 0 0 1 1 1 -1 -1

1 = long (holding buy), -1 = short (holding sell), 0 = flat (no position)

STEP 7 — Calculate Returns

# Buy-and-hold: daily % change in price
data['bh_ret'] = data['Close'].pct_change()
# = (today - yesterday) / yesterday

# Strategy return = position × daily return
# MUST use .shift(2) — see Section 1 box!
data['strat_ret'] = (
    data['position'].shift(2) * data['bh_ret']
)

⚠️ Without .shift(2): You use the signal day's position for that same day's return — look-ahead bias. Backtest looks great, real trading fails.

STEP 8 — Equity Curve (Compounded)

# Returns COMPOUND — must multiply, not add
# (1+r1) × (1+r2) × ... = $1 growth over time
data['bh_equity'] = (
    (1 + data['bh_ret']).cumprod()
)
data['strat_equity'] = (
    (1 + data['strat_ret']).cumprod()
)
# Get total return as a percentage:
bh = (data['bh_equity'].iloc[-1] - 1) * 100
st = (data['strat_equity'].iloc[-1] - 1) * 100
print(f"Buy & Hold: {bh:.2f}%")
print(f"Strategy:   {st:.2f}%")

💡 cumprod() not cumsum(): 5% + 5% ≠ 10.25%. Returns compound. Use (1+r).cumprod().

STEP 9 — Square Off + Deep Analysis

# Best practice: force-close the last position
last = data.tail(1).copy()
last['signal']   = -1  # trigger exit
last['position'] =  0  # no more position
data = pd.concat([data, last])

# ── Deep Analysis with quantstats ──────
import quantstats as qs
# Full report: Sharpe, drawdown, monthly
# returns heatmap, rolling volatility, etc.
qs.reports.full(data['strat_ret'])

quantstats gives you:

• Sharpe ratio, Sortino ratio

• Max drawdown + recovery time

• Monthly returns heatmap

• Cumulative & rolling charts

7️⃣ SIMPLE RETURNS vs LOG RETURNS — When to Use Which

Side-by-Side Comparison

FeatureSimple ReturnLog Return
Formula(P₁ − P₀) / P₀ln(P₁ / P₀)
Python code.pct_change()np.log(P/P.shift(1))
Combine over timeMultiply: (1+r)×(1+r)Just add: r1+r2+r3
Compounding code(1+r).cumprod()r.cumsum()
Normal dist?NoYes ✅
Daily differenceTiny for small moves (1-2%) — nearly identical in a chart
ComputationHeavier (multiply)Lighter (add)

💡 When the difference becomes big: Stock drops from $100 to $10 (−90% crash):
Simple return = −0.90 = −90%
Log return = ln(10/100) = −2.30 = −230%
For small daily moves the gap is invisible. It only matters for large price swings.

Code + Decision Guide

# Simple returns (used in this strategy)
data['simple'] = data['Close'].pct_change()
# Equity: (1 + data['simple']).cumprod()

# Log returns (for statistical models)
data['log'] = np.log(
    data['Close'] / data['Close'].shift(1)
)
# Equity: data['log'].cumsum()  ← just add them!

Use SIMPLE when:

• Reporting strategy returns clearly ("up 5%")

• Short-term discrete periods

• Portfolio return calculations

• This lecture / DMP strategies

Use LOG when:

• Building statistical / ML models

• Assuming normal distribution of returns

• Long-term compounding analysis

• Forecasting price models

In practice: EPAT courses use a mix of both. For strategy backtesting (like DMP-01), simple returns are standard. For advanced modeling and statistics (later in the course), log returns are preferred. When in doubt, check the course notebook.

8️⃣ VECTORIZED vs EVENT-DRIVEN BACKTESTING — Two Ways to Do the Same Thing

✅ VECTORIZED (What We Did in Section 6)

# ALL 899 rows processed at once — like array * 3
data['signal'] = np.where(
    (data['SMA'] > data['LMA']) &
    (data['SMA_prev'] < data['LMA_prev']),
    1, 0
)
# One line, all rows checked simultaneously
SpeedVery fast ⚡
Code styleShort and clean
Best forPrototyping indicator-based strategies
Cannot doOrder management, slippage, real-time reaction

⚠️ EVENT-DRIVEN (Row-by-Row with for Loop)

trades = []
for i in range(len(data)):   # One row at a time
    sma = data['SMA'].iloc[i]
    lma = data['LMA'].iloc[i]
    if sma > lma:            # Check condition THIS row
        trades.append({
            'date':  data.index[i],
            'price': data['Close'].iloc[i],
            'side':  'long'
        })
# Same result — but processes one row at a time
SpeedSlower 🐢 (row-by-row)
Code styleLonger, more explicit
Best forIntraday, tick data, live trading
Can doRealistic slippage, fill handling, cancellation logic

DECISION GUIDE — Which One Do I Use?

Testing a new strategy idea?

→ Vectorized

Daily or weekly indicators?

→ Vectorized

Intraday / tick-by-tick?

→ Event-Driven

Live trading system?

→ Event-Driven (DMP-04)

9-STEP CHECKLIST

1️⃣ Download data (multi_level_index=False)

2️⃣ Calculate SMA (21d) + LMA (63d)

3️⃣ Shift both → SMA_prev, LMA_prev

4️⃣ dropna(inplace=True)

5️⃣ Signals: np.where() twice (1 and −1)

6️⃣ Positions: replace(0, ffill)

7️⃣ bh_ret = pct_change() → strat_ret = position.shift(2) × bh_ret

8️⃣ Equity: (1+ret).cumprod()

9️⃣ Square off last position

🎯 KEY TAKEAWAYS

1. Always add multi_level_index=False when downloading from Yahoo Finance — without it your column names break.

2. .shift(2) on position is non-negotiable in daily backtests — skip it and your backtest is lying to you (look-ahead bias).

3. Vectorized = fast for prototyping. Event-driven = realistic for live trading. Start vectorized, graduate to event-driven.

4. Use (1+r).cumprod() for equity curves — returns compound, they don't add up.

THE 9-STEP RECIPE: Download → SMA+LMA → Shift(1) for prev → dropna → Signals (np.where) → Positions (ffill) → Returns (shift(2) × pct_change) → Equity (cumprod) → Square Off

THE ONE RULE TO NEVER BREAK: You cannot trade at the price that generated your signal. Signal at close Day N → trade at close Day N+1 → return measured on Day N+2. That's the shift(2).
