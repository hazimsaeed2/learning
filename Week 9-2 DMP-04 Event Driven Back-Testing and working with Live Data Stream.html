Event-Driven Back-testing & Live Data Streams

Instructor: Dr. Yves Hilpisch — refactor the vectorized back-test into OOP, event-driven code that also runs live.

THE PROBLEM
Vectorized back-tests treat each bar independently — no order queues, partial fills, latency or path-dependence.
YOUR GOAL
Model a stream of events so the back-test mirrors a real trading system, one event at a time.
THE EDGE
Encapsulated "Lego brick" classes → the back-test engine and the live engine are the same code.
REALITY CHECK → ~70–77% of retail CFD accounts lose money (CMC 76.6%, IG 71%, FXCM 69%); even hedge funds rarely beat a simple SPY buy-and-hold long-term (the curse of big operations). No shortcuts — finance + math + Python are one skill set.

2 · The four events

Each event = a timestamp + a description of what happened. They flow through a single queue.

@dataclass
class MarketEvent(Event): price        # new bar
class SignalEvent(Event): direction    # +1/-1/0
class OrderEvent(Event):  quantity     # trade size
class FillEvent(Event):   quantity,price # done

Market → Signal → Order → Fill

3 · The four Lego bricks

  • DataHandler — streams prices → MarketEvents.
  • Strategy — MarketEvent → SignalEvent (when conditions met).
  • Portfolio — signal → OrderEvent; tracks position, cash, equity (mark-to-market).
  • ExecutionHandler — OrderEvent → FillEvent.

Each class reacts to one event type. Swap any brick without touching the rest.

4 · The engine = queue + event loop

q = deque()
while bar := data.stream_next():
    q.append(bar)
    while q:
        ev = q.popleft()           # FIFO
        if   isinstance(ev,MarketEvent): strat...; port...
        elif isinstance(ev,SignalEvent): port.on_signal()
        elif isinstance(ev,OrderEvent):  exe.on_order()
        elif isinstance(ev,FillEvent):   port.on_fill()

5 · One engine, different hyper-parameters

Same event-driven engine, three runs: 1-lag momentum loses (−10%); lags=7 + threshold=0.01 beats buy & hold (+86% vs +59%). The result is dominated by the hyper-parameters — a loud warning about over-fitting.

6 · Momentum strategy + two knobs

Time-series momentum: if recent return > 0 → go long, else short (tied to autocorrelation / AR).

  • lags — days of momentum (1 = AR(1), or 3/5/7…).
  • threshold — ignore tiny moves to cut turnover; a true hyper-parameter, "as important as the idea" (1.75 good, 0.01 or 2.5 worse in the lecture).
if abs(mom) < threshold: return     # no trade
direction = 1 if mom > 0 else -1

8 · Going live = swap ONE brick

The pay-off of the architecture: to trade live, replace only the CSVDataHandler with a streaming feed. The Strategy, Portfolio, Execution and the event loop stay identical — back-test code == live code.

class LiveDataHandler:                 # same interface
    def stream_next(self):
        tick = broker.get_next_tick()    # websocket / REST
        return MarketEvent(time=tick.time, price=tick.price)
# BacktestEngine(LiveDataHandler(), strat, port, exe).run()  # live!
💡 Tips
  • Events are dataclasses — timestamp + payload.
  • One component appends events; others consume them.
  • Keep the code base small — changes stay simple (Hilpisch debugged a missing minus sign live!).
⚠ Traps
  • Hyper-parameters (lags, threshold) can flip a strategy from −10% to +86% → over-fitting risk.
  • Single in-sample run flatters; use rolling / out-of-sample windows.
  • A pretty total return can hide a 970-day (multi-year) drawdown.

30-second glossary

event-driven — process one event at a time (vs whole arrays).
encapsulation — bundle data + behaviour in a class.
MarketEvent — a new price bar.
SignalEvent — strategy's long/short/flat decision.
OrderEvent — a request to trade a quantity.
FillEvent — confirmation a trade executed.
DataHandler — streams prices; swap for live feed.
Portfolio — tracks position, cash, equity.
ExecutionHandler — turns orders into fills.
event loop — pop + route events from a deque.
mark-to-market — value position at the latest price.
hyper-parameter — a knob (lags, threshold) that defines behaviour.
