CORE DOUBT — `df.shape` vs `df.describe()`

Why no parentheses on one but not the other? An attribute is a stored value; a method is an action you call.

expressionkindparens?result
df.shapeattribute (a tuple)no ()(2460, 7)
df.describe()method (callable)needs ()count/mean/std…
df.shape()attribute mis-called—'tuple' not callable ✗
Syntax rule: library.method(arguments) — parentheses hold arguments. No arguments to pass + just reading a fact ⇒ it's an attribute, no parens.

Which bracket goes where?

( )
round
  • call functions/methods
  • pass arguments
  • define tuples (immutable)
  • group arithmetic
def f(a,b): return (a+b, a*b)
[ ]
square
  • define lists (mutable)
  • index: lst[0]
  • slice: lst[0:3]
lst = [1,5.9,True]
lst[0:3]  # 3 excluded
{ }
curly
  • define sets (unique)
  • define dicts key:value
  • vars in f-strings
d = {"MSFT":300}
f"price {d['MSFT']}"
comma , separates items/args (like English) · underscore _ makes long names readable (my_first_var).

The five ways to import

formwhen
import mathplain
import numpy as nppreferred — alias saves typing
from math import *discouraged — floods namespace
from math import sqrtjust one function (heavy libs)
from math import sqrt as rootfunction + alias
⚠ order: from B import A ✓   import A from B ✗ — the library (stored thing) comes first.
5

Namespace overriding

The danger of import *: same name in two libraries → the later import wins.

from math import sin   # real
from cmath import sin  # OVERRIDES -> complex
sin(1.0)  # (0.841+0j)  cmath won
💡 Fix: keep names explicit — math.sin(1.0) — so anyone can trace where a function came from.
Working directory: os.getcwd() to check · os.chdir(path) to move · read_csv only sees files in the CWD · dir(lib) lists contents · name? shows the docstring.
